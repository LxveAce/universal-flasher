# Suicide Marauder — Research Digest

> Generated from the deep-research fan-out (46 agents, 8 dimensions, adversarial verification + completeness critic). This is the evidentiary basis for ARCHITECTURE.md / THREAT-MODEL.md / SAFETY.md. Each high-risk claim was adversarially verified; verdicts are shown.

## Completeness Critic — Recommended Architecture

STRATEGY A (fork Marauder, Guardian-in-factory) is the architecture the evidence best supports, and it is now confirmed against the live local source. Build a CUSTOM partition table with a real `factory` slot for Guardian + `ota_0` for Marauder + `nvs` (Marauder) + a SEPARATE small `nvs`/data partition for Guardian config + `otadata` + spiffs. Target 8MB minimum, prefer 16MB: two ~1.875MB app slots plus filesystems overflow 4MB, so a 4MB Guardian+Marauder build must be treated as impossible. On cold boot with an erased (all-0xFF) otadata, the 2nd-stage bootloader runs Guardian (factory); Guardian validates its gate, then `esp_ota_set_boot_partition(ota_0)` + `esp_restart()` hands off to Marauder — Espressif's documented factory->OTA transition. CRITICAL collision (verified in SDInterface.cpp lines 217-223): Marauder's own SD updater already calls esp_ota_get_next_update_partition + esp_ota_set_boot_partition, so Guardian must use the factory partition explicitly and re-arm by either erasing otadata or set_boot_partition(factory); additionally enable CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE so Marauder boots PENDING_VERIFY and auto-reverts to the prior-valid slot if it never marks itself valid. Strategy B (separate official binary) is rejected: it duplicates all drivers and collides head-on with that existing OTA chainload. The Guardian gate is bolted in as a new BootGate.h/.cpp called in setup() AFTER display_obj.RunSetup() (line 312) and BEFORE settings_obj.begin() (line 348) — confirmed hook region in the local source.

ARMING SWITCH fails safe via TWO-FACTOR, which resolves the contradiction between the dead-man design and the fail-safe-default philosophy: the device must be EXPLICITLY ARMED (a deliberate provisioned/persisted state, default DISARMED) AND the dead-man line read LOW for a wipe to be possible. The arming line itself uses INPUT_PULLDOWN (the existing Switches.cpp already supports INPUT_PULLDOWN, line 27) wired so only the intact/sealed/armed switch drives the pin HIGH; a cut/unplugged/floating wire reads LOW. Thus, when armed, removing/cutting the switch triggers wipe; but a freshly-flashed, un-provisioned, or unknown-state device is DISARMED and physically cannot wipe. Default pin: GPIO27 (classic ESP32/CYD) or Grove G2 (S3 boards); never a strapping pin (0/2/12/15 classic; 0/3/45/46 S3; 2/8/9 C3); GPIO34-39 are input-only and need an external 10k pull-down. Read at boot in setup() after a ~10ms settle, sample 8+ times, require unanimous ARMED. All confirmation/'are you sure' friction lives at CONFIGURATION/provisioning time; once legitimately triggered the wipe is fast and non-abortable.

WIPE ORDERING (best-effort, honestly documented — there is NO runtime crypto-erase on ESP32, the AES key eFuse is HW read+write-protected): build the Guardian/brick image with CONFIG_SPI_FLASH_DANGEROUS_WRITE_ALLOWED=y. Stage 1: esp_partition_erase_range over Marauder ota_0, Guardian config nvs, Marauder nvs, spiffs/SD (overwrite SD via the SPI/SdFat path while inserted; full-range single pass on NOR + native card->erase()), and otadata. Stage 2 (the only genuinely high-risk, UNVERIFIED step): a final routine that raw-erases the partition table (0x8000), bootloader (0x1000 classic / 0x0 S3/C3), and the factory/app region — boot chain dies last. Because this self-erase-of-running-app is undocumented, treat the exact IRAM/cache mechanics as a hardware spike, NOT a settled recipe. For real anti-forensic value, this MUST be paired with Release-mode Flash Encryption + Secure Boot v2 (so a pulled/powered-off device isn't trivially read, the erase destroys ciphertext, AND UART download is locked so the brick is actually unrecoverable rather than re-flashable).

FLASHER PROVISIONING: one esptool write_flash pass with the pair list bootloader + custom partitions.bin(0x8000) + boot_app0(0xe000) + guardian_factory(0x10000) + marauder(ota_0) + nvs_config(nvs offset) + otadata_FF(otadata offset), reusing the existing flasher.flash() shape. Per-device config (arm GPIO, armed level, max attempts, wipe flags) is baked into an NVS binary via nvs_partition_gen; the password is hashed ON THE HOST at flash time (PBKDF2-HMAC-SHA256 high-iteration as the agreed KDF, since Argon2id's 19MiB OWASP minimum is impossible on ESP32 RAM and argon2-cffi isn't installed) with a fresh os.urandom salt — only {salt, hash, KDF params} written to NVS, plaintext zeroized, never logged, never a CLI arg. Guardian re-derives with the identical KDF. First boot forced to Guardian via the host-generated 0xFF otadata blob (read the real offset from the partition table, do NOT hardcode 0xe000 against a stock IDF table). UX: a single 'Suicide' checkbox swapping the plain-Marauder app for the bundle, prompting only for password + arm pin/level. Note flash encryption does NOT cover the NVS data partition; the salted hash is safe in plaintext NVS, but if arm-pin/level must be hidden, add NVS encryption + nvs_keys partition.

PER-HARDWARE TIERING (gated by a compile-time INPUT_CLASS flag mirroring HAS_TOUCH/HAS_MINI_KB/MARAUDER_CARDPUTER/HAS_BUTTONS/headless): FULL on-device gate for touchscreen (reuse keyboardInput), Mini joystick (miniKeyboard with do_pass=true — confirmed present), and Cardputer (native QWERTY, strongest); WEAK button-combo gate or host-assisted for M5StickC (2 usable buttons); SERIAL/host-assisted only for headless boards (add an 'unlock <pass>'/'wipe' CLI command — confirmed the firmware reads only Serial there). Self-destruct is standalone-capable on every class (pure firmware on internal flash); the removable SD is the one evidence gap, mitigated only by at-rest encryption.

### Gaps
- WIPE LATENCY / POWER MODEL UNQUANTIFIED: No research dimension measures how long the full destruction sequence takes on the actual target chip, nor whether the ESP32 retains power long enough to finish. Crypto-erase prior art (BusKill ~5s) is fast precisely because it destroys a small key — but the ESP32 dimensions concluded a true runtime crypto-erase by key destruction is IMPOSSIBLE (eFuse AES key is HW read+write-protected). That means 'wipe' = bulk erase of partitions + boot chain, which on a 16MB part can take many seconds to a minute (in-place encryption alone is documented at ~1 min). The snatch-the-device-while-powered scenario is never reconciled against this. This is the single biggest unanswered design question: the panic primitive everyone else uses (key destroy) is the one ESP32 cannot do.
- TARGET HARDWARE NEVER FIXED: Every dimension lists 'which exact board/chip/flash size' as an open question and none resolves it. Flash size (4MB vs 8/16MB) decides whether Guardian-in-factory even fits; chip family (ESP32 vs S3/C3) decides bootloader offset (0x1000 vs 0x0), arming-pin map, and SD interface. The whole architecture is conditional on a fact never pinned down.
- SECURE BOOT / FLASH ENCRYPTION DECISION UNRESOLVED AND LOAD-BEARING: Multiple dimensions note that without Secure Boot, the entire boot-gate is cosmetic — an attacker just reflashes past Guardian via esptool. Without it, 'unrecoverable brick' is also false (UART download survives erase and re-flashes). Yet no dimension commits to whether Release-mode Secure Boot + Flash Encryption is in scope. The gate's anti-forensic value is binary on this and it is left open.
- RECOVERY/ESCROW MECHANISM NOT DESIGNED FOR ESP32: The duress-systems dimension makes owner-only out-of-band recovery a hard requirement (Kali luksHeaderBackup pattern), but no dimension maps this onto the ESP32 design. If the gate holds a wrapped key, where is the escrow? If the panic action is a brick, the 'recovery' is just host re-flash (no owner secret needed) — but then the device's stored data is genuinely gone with no owner recovery path, which contradicts the 'recoverable by owner' requirement. The conflict between 'real unrecoverable wipe' and 'owner-recoverable' is never resolved for this product.
- ON-DEVICE KDF PARAMS NOT SETTLED: OWASP Argon2id minimum (19 MiB) is impossible on ESP32 RAM; PBKDF2-high-iteration is the fallback, but the exact agreed params (and the brute-force resistance trade-off that implies) are left as an open question. Host and device must use identical KDF+params; this contract is undefined.
- WHAT THE GATE ACTUALLY CUSTODIES IS AMBIGUOUS: The duress dimension explicitly flags that it's unclear whether the ESP32 gate holds/releases a disk-unlock key for a host (HSM-style), gates power/boot to a host, or just protects its own internal flash/SD. The other dimensions implicitly assume self-contained (protect own flash + SD). These are very different products and the scope is never nailed down.
- NO TEST/SAFE-MODE DESIGN: The duress dimension makes a SAFE/dry-run mode (full trigger chain against a DUMMY key) a REQUIREMENT, but no firmware/flasher dimension specifies how to build it (e.g., a build flag that points the destroy routine at a scratch partition). It is required but unscoped.
- RTC/PSRAM/SD-pulled-before-trigger DATA REMNANTS: Self-wipe dimension flags that data in RTC slow memory, PSRAM, and a removed SD card is out of scope of internal-flash wipe; the SD dimension confirms a pulled card is unreachable. The only mitigation offered (encrypt SD at rest) depends on flash encryption being enabled — circling back to the unresolved Secure Boot/FE decision.
- BROWNOUT / LOW-BATTERY BOOT POLICY: Arming dimension flags that on battery boards (StickC/Cardputer) an undervoltage boot could spuriously read the arming line; whether a low-battery boot should fail armed or disarmed is raised but not decided — and it directly conflicts with the user's stated reliability-first stance.

### Contradictions (and how synthesis resolves them)
- CRYPTO-ERASE IS 'THE UNIVERSAL MECHANISM' (duress dimension) vs 'RUNTIME CRYPTO-ERASE IS IMPOSSIBLE ON ESP32' (self-wipe dimension). The duress-systems research concludes crypto-erase (destroy the key, not the data) is the converged, fast, NIST-standard primitive and recommends REQ-2 'implement wipe as destruction of the small key/header material — never bulk overwrite.' The ESP32 self-wipe dimension proves the opposite for this hardware: the AES flash-encryption key lives in a HW read- AND write-protected eFuse that software can neither read nor overwrite, so there is NO runtime key-destroy primitive. The product therefore CANNOT use the recommended panic mechanism; it must fall back to bulk erase + boot-chain brick, which is exactly what the duress dimension warns against (slow, conflicts with snatch-resistance). This is the central unresolved tension across the whole corpus.
- FAIL-SAFE DEFAULT = LEAST-DESTRUCTIVE / DISARMED (duress dimension REQ-1, REQ-3: 'a freshly-flashed or unknown-state device must be physically incapable of crypto-erasing real data; require BOTH arming AND trigger') vs ARMING-LINE FAIL-SAFE = 'absence of signal = DISARMED = WIPE' (arming-switch dimension). The arming dimension's dead-man design makes a cut/floating/unpowered wire TRIGGER the wipe (fail-toward-destruction). The duress dimension's design philosophy is fail-AWAY-from-destruction (default disarmed, destruction needs deliberate positive intent). These are directly opposed fail-safe philosophies. They can be reconciled (the dead-man line should gate only when the system is ALREADY in an explicitly-armed state — i.e. two-factor: armed-flag AND dead-man-low), but as written the two dimensions describe contradictory default behaviors and the synthesis must explicitly pick the two-factor reconciliation.
- 'SINGLE-PASS OVERWRITE IS SUFFICIENT ON FLASH' (self-wipe dimension, medium confidence) vs the SD-card verification's CORRECTION that the UCSD study found whole-disk overwrite of even the VISIBLE LBA space 'usually but not always' needed two passes and one drive retained ~1% after twenty passes. The internal-flash dimension asserts one pass is forensically sufficient and 'multi-pass is theater'; the SD dimension's own cited primary source partially refutes the one-pass-is-sufficient framing for flash/FTL media. Net: 'one pass' is defensible for NOR internal flash but NOT a safe blanket claim for the managed-NAND SD card, and the corpus states it both ways.
- otadata DEFAULT OFFSET stated as 0xe000 in the flasher dimension's claim/recommendation vs verification finding the ESP-IDF DEFAULT is 0xd000 (and 0xe000 is the Marauder-specific value because Marauder enlarges nvs to 0x5000). The flasher's 'flash an all-0xFF blob at the otadata offset' recommendation hardcodes 0xe000; if reused against a stock ESP-IDF table it lands mid-partition. The corpus contains both the wrong-for-IDF 0xe000 and the corrected 'read offset from the actual partition table' guidance.
- SELF-ERASE ORDERING needs IRAM_ATTR + manually-disabled scheduler/interrupts (self-wipe claim 'Safe destruction ORDER') vs the verification of that same claim, which rated it UNCERTAIN because ESP-IDF docs say esp_flash_erase_region ALREADY disables caches/suspends tasks/disables non-IRAM-safe interrupts automatically and transparently — so manually disabling the scheduler is unnecessary and the self-erase-of-running-app is undocumented. The corpus simultaneously recommends and undercuts the manual-IRAM-orchestration approach.
- min_spiffs 'spiffs = 192K, no coredump' (integration A/B dimension) vs the 'boot flow' dimension's verbatim CSV showing spiffs=0x20000 (128K) PLUS a coredump partition. This is a version skew (192K/4-entry = arduino-esp32 core 1.0.6; 128K+coredump = current 2.0.x/3.x). Two dimensions describe the stock layout differently; current arduino-cli tooling matches the 128K+coredump version, so the 192K figure is stale.

## ESP32 boot flow & partition architecture for a Guardian app gating ESP32 Marauder

**Summary.** The ESP32 boot chain is: ROM (1st-stage) bootloader -> 2nd-stage bootloader (in flash) -> it reads the partition table at 0x8000, consults the otadata partition, and jumps to the selected app. A "Guardian gates Marauder" design maps cleanly onto Espressif's native factory+OTA model: put Guardian in the `factory` slot and Marauder in `ota_0`. Guardian validates whatever conditions you want, then calls esp_ota_set_boot_partition(ota_0) + esp_restart() to hand off into Marauder — this is exactly the documented factory->OTA transition. otadata (a 0x2000 / two-sector partition) records which slot boots; when otadata is blank/invalid the bootloader falls back to the factory app (Guardian), which is what makes the gate reassert itself. The hard constraint is flash size: Marauder's own app is ~1.9 MB (it ships with the Arduino "Minimal SPIFFS Large APPS with OTA" / min_spiffs scheme, app0=0x1E0000), so a Guardian(factory ~1.9MB) + Marauder(ota_0 ~1.9MB) + otadata + nvs + spiffs layout does NOT fit in 4 MB — you need 8 MB minimum, comfortably 16 MB. A critical chip-dependent gotcha: the 2nd-stage bootloader is flashed at 0x1000 on ESP32/ESP32-S2 but at 0x0 on ESP32-S3/C3/C6/H2.

### Findings

**[high] ESP32 boot is a multi-stage chain: ROM 1st-stage bootloader -> 2nd-stage bootloader in flash -> partition table -> selected app.**

The masked ROM 1st-stage bootloader loads the 2nd-stage bootloader from flash into RAM. The 2nd-stage bootloader reads the partition table (default offset 0x8000), then for an app-with-OTA config consults the otadata partition to decide which app slot to load, verifies it, and jumps to it. On ESP32-S3 the docs state plainly: 'the binary image which resides at offset 0x0 in flash is the second stage bootloader' and 'Second stage bootloader binary image is loaded from the start of flash at offset 0x0.'

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/bootloader.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32s3/api-guides/startup.html

**[high ⚠HIGH-RISK] 2nd-stage bootloader flash offset is CHIP-DEPENDENT: 0x1000 on ESP32 & ESP32-S2; 0x0 on ESP32-S3, C3, C6, H2.**

Original ESP32 and ESP32-S2 flash the bootloader at 0x1000, leaving the first 4 KB for ROM/secure-boot use. Newer chips (S3, C3, C6, H2) place the bootloader at 0x0. The partition table offset stays 0x8000 across all of them (configurable via CONFIG_PARTITION_TABLE_OFFSET). This matters for the Guardian flasher: a flash command like `esptool write_flash 0x0 bootloader.bin ...` is correct for S3/C3 but WRONG for classic ESP32 (must be 0x1000). Marauder hardware spans both families (classic ESP32 WROOM boards AND ESP32-S3 boards like the Cardputer/Marauder Mini), so the toolchain must branch on chip.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32s3/api-guides/startup.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/bootloader.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32c3/api-guides/bootloader.html

**[high] Partition table: flashed at 0x8000, length 0xC00 (+MD5), occupies one 0x1000 sector. CSV columns are Name, Type, SubType, Offset, Size, Flags.**

Name <=16 bytes incl. null terminator. Type: app or data (or numeric 0-254; 0x00-0x3F reserved for ESP-IDF). Offset/Size accept decimal, 0x hex, or K/M multipliers (K=1024, M=1024*1024). Blank offset auto-advances after the previous partition with alignment. Flags supported: `encrypted`, `readonly`. App partitions MUST be aligned to 0x10000 (64 KB); an unaligned app offset is a hard error from gen_esp32part.py. Data partitions only need 4 KB (0x1000) sector alignment.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/partition-tables.html

**[high] App subtypes: factory(0x00), ota_0..ota_15 (0x10-0x1F), test(0x20). Data subtypes include ota(0x00), phy(0x01), nvs(0x02), coredump(0x03), spiffs(0x82), fat(0x81).**

For the Guardian design: Guardian = type app, subtype factory; Marauder = type app, subtype ota_0. otadata = type data, subtype ota, size 0x2000. A small config/credential store = type data, subtype nvs (or a custom data subtype). Filesystem = type data, subtype spiffs (or littlefs via the fat/spiffs slot; Marauder uses spiffs).

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/partition-tables.html

**[high] otadata is 0x2000 (two flash sectors); it is what the bootloader reads to pick the boot app, with power-loss-safe dual-sector + sequence-number + CRC design.**

Docs: otadata 'should be 0x2000 bytes' and the bootloader 'consults this data in order to know which app to execute.' The two sectors are independently erased/written with matching data; 'if they disagree a counter field is used to determine which sector was written more recently.' Boot-selection fallback rules: if otadata is empty/invalid, the bootloader boots the factory app if present (this is the Guardian-reassert mechanism); otherwise it boots ota_0. Slots flagged ESP_OTA_IMG_INVALID or ESP_OTA_IMG_ABORTED are skipped.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/system/ota.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/partition-tables.html

**[high] A factory app CAN validate, then set boot to an OTA partition and reboot into it — this is documented and is the exact Guardian->Marauder handoff.**

esp_err_t esp_ota_set_boot_partition(const esp_partition_t *partition) writes otadata to select the next boot slot; after ESP_OK, esp_restart() boots that slot. The factory (Guardian) finds the Marauder partition via esp_partition_find_first(ESP_PARTITION_TYPE_APP, ESP_PARTITION_SUBTYPE_APP_OTA_0, NULL), calls set_boot_partition on it, then esp_restart(). esp_ota_get_running_partition() returns the currently-executing app; esp_ota_get_boot_partition() returns the slot configured to boot next. Returns ESP_ERR_OTA_VALIDATE_FAILED under secure boot if the target image fails verification.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/system/ota.html

**[medium ⚠HIGH-RISK] To make the gate reassert (return to Guardian after Marauder), the design relies on factory fallback OR an explicit re-point of otadata back to factory.**

There is no 'boot factory' subtype value to pass to set_boot_partition for the next boot persistently except by selecting the factory partition object, OR by erasing otadata so the bootloader falls back to factory. Practical patterns: (a) Marauder (or a watchdog/RTC condition) calls set_boot_partition(factory) before reset; (b) Guardian erases otadata after each handoff so a cold boot always lands in Guardian; (c) optional rollback: with CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE a freshly-booted OTA app starts in ESP_OTA_IMG_PENDING_VERIFY and is auto-rolled-back to the previous (Guardian) app unless it calls esp_ota_mark_app_valid_cancel_rollback(). This is a strong gating primitive: if Marauder never marks itself valid, the bootloader reverts to Guardian on the next reset.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/system/ota.html

**[high] Marauder's app is ~1.9 MB and it ships with the Arduino 'Minimal SPIFFS (Large APPS with OTA)' (min_spiffs) scheme.**

Marauder wiki/build docs specify the Arduino partition scheme 'Minimal SPIFFS (Large APPS with OTA)'. That scheme (min_spiffs.csv) gives app0=app1=0x1E0000 (1.875 MB, '1900K') with only 0x20000 (128 KB) spiffs. Sources: https://github.com/justcallmekoko/ESP32Marauder/wiki/installing-firmware-from-source, https://docs.espressif.com/projects/arduino-esp32/en/latest/tutorials/partition_table.html

**[high ⚠HIGH-RISK] Two ~1.9 MB apps (Guardian factory + Marauder ota_0) do NOT fit in 4 MB. Need 8 MB minimum; 16 MB is comfortable.**

4 MB budget = 0x400000 (4,194,304 B). Fixed overhead before apps: bootloader+gap to 0x8000 (32 KB) + partition table 0x1000 (4 KB) + nvs ~0x5000 (20 KB) + otadata 0x2000 (8 KB) + phy/config ~0x1000. First app must start at 0x10000 (64 KB). Two 0x1E0000 app slots alone = 0x3C0000 (3.75 MB); starting the first at 0x10000 the second ends at 0x10000+0x3C0000 = 0x3D0000, leaving only 0x30000 (192 KB) for ALL of otadata+nvs+spiffs+coredump+phy. That is why min_spiffs on 4 MB only fits ONE large app + ITS sibling with a tiny 128 KB spiffs and NO factory slot. Adding a true 3rd full-size app (a real Guardian factory slot) is impossible on 4 MB. On 8 MB (0x800000) you can fit factory(0x1E0000)+ota_0(0x1E0000) = 0x3C0000 and still have ~4 MB for nvs/otadata/config/spiffs. 16 MB removes all pressure and is what most modern Marauder S3 boards (Cardputer, Marauder v6/Mini) actually carry.

### Recommendations
- Use a CUSTOM partition CSV with a real `factory` slot for Guardian + `ota_0` for Marauder. The stock Arduino min_spiffs/default schemes have NO factory partition (only app0/app1=ota_0/ota_1), so they cannot express 'factory gates OTA' as-is.
- Target 8 MB flash minimum; prefer 16 MB. Marauder's app is ~1.9 MB (min_spiffs app0=0x1E0000); two full-size apps + otadata/nvs/spiffs overflow 4 MB. Do not promise a 4 MB Guardian+Marauder build — quantify it as impossible for two large apps.
- Boot/handoff flow: Guardian (factory) runs first on cold boot (blank otadata), validates its gate condition, then esp_partition_find_first(APP, OTA_0) -> esp_ota_set_boot_partition() -> esp_restart() to enter Marauder. To re-arm the gate, either erase otadata or set_boot_partition(factory) before reset.
- Branch the flasher on chip family for bootloader offset: 0x1000 for classic ESP32 / ESP32-S2; 0x0 for ESP32-S3 / C3 / C6 / H2. Partition table stays at 0x8000. Get this wrong and the device bricks/won't boot.
- Consider the rollback primitive as a hard gate: enable CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE so Marauder boots in PENDING_VERIFY; if it never calls esp_ota_mark_app_valid_cancel_rollback(), the bootloader auto-reverts to Guardian on next reset — a clean way to force the gate to re-assert without trusting Marauder to behave.
- Add a dedicated small `data,nvs` (or custom data subtype) partition for Guardian config/credentials, kept separate from Marauder's own nvs/spiffs so a Marauder factory-reset or spiffs format cannot wipe the gate state.
- If you ever need to swap Marauder versions, keep ota_1 too (A/B) — but on 8 MB that means factory(1.9) + ota_0(1.9) + ota_1(1.9) = 5.7 MB of apps, leaving ~2 MB for fs; on 16 MB it's trivial. Drop ota_1 if you only need Guardian+single Marauder.
- Use esp_ota_get_running_partition() inside both apps so each can detect 'am I Guardian or Marauder' at runtime and behave accordingly (e.g., Marauder offering a 'return to Guardian' menu item).

## Secure data destruction of ESP32 internal flash from within running firmware (Guardian self-wipe / brick)

**Summary.** Firmware can erase and overwrite flash at runtime using esp_partition_erase_range/esp_partition_write (partition-bounded) or esp_flash_erase_region/esp_flash_write (raw address). By default IDF BLOCKS the firmware from touching the region that holds the running app, the bootloader, and the partition table via the SPI_FLASH_DANGEROUS_WRITE check, which aborts() by default; you must set CONFIG_SPI_FLASH_DANGEROUS_WRITE_ALLOWED to wipe those regions. The deepest correct CRITICAL nuance: a partition CANNOT safely erase itself while executing XIP from it, because flash erase requires the CPU cache to be disabled, so the wipe routine and all data it touches must live in IRAM/DRAM — the standard pattern is to wipe everything else first, then erase your own app + partition table + bootloader LAST from an IRAM-resident routine so the chip no longer boots (a true brick until host re-flash via esptool). On the encryption side, a true instant "crypto-erase" by destroying the key is NOT available at runtime: the AES key in the flash_encryption/BLOCK_KEYN eFuse is hardware read- AND write-protected, so software can neither read nor overwrite it. What firmware CAN do is burn FLASH_CRYPT_CNT (in dev mode only) or burn other eFuses via esp_efuse_write_field_* — all one-way and irreversible — but this does not erase the key and is risky. Because NOR flash has no magnetic remanence, a single overwrite pass (or a single sector erase to 0xFF) is forensically sufficient; multi-pass is theater.

### Findings

**[high] Two API layers exist: partition-scoped (esp_partition_erase_range / esp_partition_write) and raw-address (esp_flash_erase_region / esp_flash_write). Both are callable from app code.**

Signatures: esp_err_t esp_partition_erase_range(const esp_partition_t *partition, size_t offset, size_t size); esp_err_t esp_partition_write(const esp_partition_t *partition, size_t dst_offset, const void *src, size_t size); esp_err_t esp_flash_erase_region(esp_flash_t *chip, uint32_t start, uint32_t len); esp_err_t esp_flash_write(esp_flash_t *chip, const void *buffer, uint32_t address, uint32_t length). Erase granularity is the 4 KB sector (chip->drv->sector_size, typically 4096) and both offset and size must be 4 KB-aligned. Before any write the region must already be erased. Locate target partitions (ota_0/Marauder app, nvs, spiffs, custom config) with esp_partition_find_first(type, subtype, label) / esp_partition_find().

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/storage/partition.html, https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/peripherals/spi_flash/index.html

**[high ⚠HIGH-RISK] By DEFAULT, IDF forbids firmware from erasing/writing the bootloader, the partition table, AND the partition that contains the currently running app. This is the central gate the Guardian must defeat.**

Mechanism: the choice CONFIG_SPI_FLASH_DANGEROUS_WRITE with options _ABORTS (default for flash builds), _FAILS, _ALLOWED. Kconfig help text: the check covers writes/erases to 'the beginning of flash (covering the bootloader and partition table) or that overlap the app partition that contains the running app.' Implementation in components/spi_flash/flash_ops.c: is_safe_write_address() delegates to esp_partition_main_flash_region_safe(); UNSAFE_WRITE_ADDRESS expands to abort() when _ABORTS is set, else returns ESP_ERR_INVALID_ARG. In the newer esp_flash layer, a write/erase overlapping a read-only partition returns ESP_ERR_NOT_ALLOWED. To wipe the bootloader/partition-table/own-app regions you MUST build with CONFIG_SPI_FLASH_DANGEROUS_WRITE_ALLOWED (it is the default ONLY for APP_BUILD_TYPE_RAM builds).

Sources: https://github.com/espressif/esp-idf/blob/master/components/spi_flash/Kconfig, https://github.com/espressif/esp-idf/blob/master/components/spi_flash/flash_ops.c, https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/peripherals/spi_flash/index.html

**[high ⚠HIGH-RISK] CRITICAL: A partition cannot reliably erase itself while the CPU is executing code XIP from it, because flash erase forces the instruction/data cache OFF. The wipe routine and everything it touches must be in IRAM/DRAM.**

From the SPI1 concurrency-constraints guide: SPI0/1 is shared between the CPU cache and the SPI1 flash peripheral, so 'the caches must be disabled while reading/writing/erasing operations,' during which 'all CPUs should always execute code and access data from internal RAM,' all other tasks are suspended and non-IRAM-safe interrupts disabled. If any code/symbol read during a flash op lives in flash cache, 'it will cause a crash.' Consequence for self-wipe: the final routine that erases the running app + partition table + bootloader must be marked IRAM_ATTR (and use DRAM/RTC data, no flash-resident constants/strings), or the CPU will fetch the next instruction from a region that is mid-erase and crash unpredictably. CONFIG_SPI_FLASH_AUTO_SUSPEND can keep cache enabled during ops but is hardware-limited (specific flash chips; not supported on all targets) and should not be relied on for a destroy path.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32s3/api-reference/peripherals/spi_flash/spi_flash_concurrency.html

**[high ⚠HIGH-RISK] Safe destruction ORDER for the Guardian brick: wipe all OTHER regions first, then self-immolate the boot chain last.**

Recommended sequence: (1) erase/overwrite the data partitions whose loss does NOT stop code execution — ota_0/Marauder app slot, nvs, spiffs/littlefs, custom config, otadata — using esp_partition_erase_range + optional esp_partition_write of esp_fill_random() data; these run fine from normal flash-resident code. (2) As the FINAL step, from a single IRAM_ATTR routine with interrupts/scheduler disabled, erase the partition table (offset 0x8000), the bootloader (offset 0x1000 on ESP32, 0x0 on S3/C3), and the running app/factory partition via esp_flash_erase_region. Once the partition table + bootloader are 0xFF, the ROM bootloader finds no valid image and the chip will not boot = true brick recoverable only by host esptool write_flash. Doing the boot chain last guarantees the earlier data wipes actually complete before the CPU loses the ground under its feet. Note: after erasing the partition table, esp_partition_* APIs are invalid — use raw esp_flash_erase_region for the last steps.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/partition-tables.html, https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/peripherals/spi_flash/index.html, https://docs.espressif.com/projects/esp-idf/en/latest/esp32s3/api-reference/peripherals/spi_flash/spi_flash_concurrency.html

**[high ⚠HIGH-RISK] Erasing the bootloader region from app code IS technically possible but only with the dangerous-write gate disabled, and it is the operation that produces the unrecoverable brick.**

With CONFIG_SPI_FLASH_DANGEROUS_WRITE_ALLOWED, esp_flash_erase_region(NULL_for_main_chip, BOOTLOADER_OFFSET, len) succeeds. There is no separate 'bootloader partition' in the partition table — it sits below the table at the start of flash — so it must be addressed by raw offset, not esp_partition_*. This is the single most load-bearing destructive write: zero/erase it last and from IRAM. If flash encryption + Secure Boot v2 are active, the bootloader itself is encrypted/signed but the erase still works (you are destroying ciphertext), and the result is equally unbootable.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/bootloader.html, https://github.com/espressif/esp-idf/blob/master/components/spi_flash/Kconfig

**[high ⚠HIGH-RISK] A true instant CRYPTO-ERASE (destroy the key to make all flash unrecoverable) is NOT available at runtime on ESP32-family. The AES flash-encryption key is hardware read- AND write-protected in eFuse.**

Espressif docs state plainly: 'The key cannot be accessed via software as the write and read protection bits for the flash_encryption eFuse are set' and 'flash encryption operations happen entirely by hardware, and the key cannot be accessed via software.' On ESP32-S3/C3 the key lives in a BLOCK_KEYN block with KEY_PURPOSE_N = XTS_AES_128_KEY (purpose value 4), 'by default protected from further writes or software readout.' Because the block is write-protected, firmware cannot overwrite/burn-over the key to obliterate it, and because it is read-protected, firmware cannot exfiltrate it either. So the magnetic-media 'crypto-erase = wipe the key, leave ciphertext' model does NOT map cleanly: you cannot delete the key at runtime. Crypto-erase is effectively only achievable physically (destroy the die) or was a host-side provisioning choice.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/security/flash-encryption.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32c3/security/flash-encryption.html

**[high ⚠HIGH-RISK] Crypto-erase does NOT require Secure Boot — but neither feature gives you a runtime key-destroy primitive. Flash Encryption and Secure Boot are independent (recommended together).**

Docs: flash encryption is 'independent' of Secure Boot; 'It is recommended to use flash encryption in combination with Secure Boot.' Practical takeaway for the Guardian: if you want flash contents to be worthless after a wipe WITHOUT relying on a key-destroy you cannot do, your security comes from actually erasing the ciphertext (the brick path above), not from a crypto-erase. Flash encryption only helps pre-brick (data at rest is ciphertext); it does not provide a faster destroy.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/security/flash-encryption.html

**[high ⚠HIGH-RISK] What firmware CAN burn at runtime: FLASH_CRYPT_CNT (dev mode only) and arbitrary unburned eFuses via esp_efuse_write_field_* — all one-way, none of which destroys the key, all of which can brick.**

Runtime eFuse write APIs exist: esp_efuse_write_field_blob(), esp_efuse_write_field_cnt(), esp_efuse_write_field_bit(), with batch staging via esp_efuse_batch_write_begin()/commit(). 'Each eFuse is a one-bit field which can be programmed to 1 after which it cannot be reverted back to 0' — irreversible. FLASH_CRYPT_CNT is 7 bits (0..127): ODD number of bits set = flash treated as encrypted (auto-decrypt on read); EVEN = bootloader treats flash as PLAINTEXT and will not decrypt, so existing ciphertext becomes 'unrecoverable garbage' and the device soft-bricks/reboot-loops. In DEVELOPMENT mode FLASH_CRYPT_CNT is not write-protected (can be toggled up to 3 times); in RELEASE mode the bootloader write-protects FLASH_CRYPT_CNT plus sets DISABLE_DL_ENCRYPT/DISABLE_DL_DECRYPT/DISABLE_DL_CACHE, so it cannot be changed at all and encryption can never be disabled. Burning FLASH_CRYPT_CNT to even is a denial-of-decrypt (soft brick), NOT a key wipe — and on a release-mode part it is impossible from firmware.

Sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/security/flash-encryption.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/system/efuse.html, https://github.com/PBearson/ESP32_Flash_Encryption_Tutorial

**[medium] Single-pass overwrite is sufficient for NOR flash; multi-pass is unnecessary. A sector erase already sets all bits to 0xFF.**

NOR flash stores bits as charge on floating gates with no magnetic remanence, so the Gutmann/multi-pass model for magnetic media does not apply. A single esp_partition_erase_range/esp_flash_erase_region sets the sector to 0xFF and physically removes the stored charge; one optional pass of esp_fill_random()-generated bytes written after erase defends against any read-before-erase residue and against trivial pattern recovery. Standard guidance: 1 erase (+ optional 1 random pass) + ciphertext-at-rest via flash encryption is the accepted bar. Caveat (not in IDF docs, flag as engineering judgment): wear-leveling layers (NVS, SPIFFS, FAT) may leave stale copies in other sectors, so for those you should erase the ENTIRE partition range, not just logically delete keys/files.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/peripherals/spi_flash/index.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/storage/partition.html

**[high] Comparison to esptool erase_flash: equivalent end-state (whole chip to 0xFF incl. bootloader/table/app) but host-side, and notably esptool REFUSES to erase when encryption/secure-boot are detected, and never touches eFuses.**

esptool erase-flash = 'full chip erase (all data replaced with 0xFF bytes)' covering bootloader, partition table, app, and user data; erase-region erases a 4 KB-aligned subrange. Key safety divergence: 'Erasing the flash chip is disabled by default if either active Secure Boot or Flash Encryption is detected' (override with --force, 'at your own risk'). esptool operates on the flash chip only; it does NOT and cannot erase the eFuse key (eFuse is separate OTP silicon), so after an esptool erase_flash on an encrypted part the key still lives in eFuse — confirming again there is no host or runtime path to 'destroy the key.' The Guardian's in-firmware brick is essentially a self-hosted erase_flash that also kills the boot chain, achieving the same unrecoverable state without a host, but it requires CONFIG_SPI_FLASH_DANGEROUS_WRITE_ALLOWED and IRAM-resident final routine that esptool does not need.

Sources: https://docs.espressif.com/projects/esptool/en/latest/esp32/esptool/basic-commands.html, https://github.com/espressif/esptool/issues/643

### Recommendations
- Build the Guardian factory/brick image with CONFIG_SPI_FLASH_DANGEROUS_WRITE_ALLOWED=y. Without it, the IDF dangerous-write check will abort() the instant you try to erase the bootloader, partition table, or your own app slot. This is the single config that gates the whole feature.
- Stage the wipe in strict order: (1) ota_0/Marauder app, nvs, spiffs/littlefs, config, otadata via esp_partition_erase_range (+ optional one esp_fill_random pass); then (2) a FINAL IRAM_ATTR routine, scheduler+interrupts disabled, that raw-erases partition table (0x8000), bootloader (0x1000 ESP32 / 0x0 on S3/C3), and the factory/app region via esp_flash_erase_region. Boot chain dies last so earlier wipes complete.
- Mark the final destroy routine IRAM_ATTR and ensure every constant/string/function it touches is in IRAM/DRAM (no flash-resident data). During flash erase the cache is off; any flash fetch crashes mid-wipe and can leave a half-bricked, partially-recoverable device.
- Do NOT design around a runtime 'crypto-erase by key destruction' — it does not exist on ESP32. The eFuse AES key is hardware read- and write-protected; firmware can neither read nor overwrite it. Get your unrecoverability from actually erasing ciphertext (the brick path), and use flash encryption only for data-at-rest protection before the wipe.
- Treat eFuse burning (esp_efuse_write_field_*) as a last-resort, irreversible accessory, not the primary mechanism. Burning FLASH_CRYPT_CNT to an even count only works in development mode and only denies decryption (soft brick) — it does not erase the key and is impossible on release-mode parts. Avoid burning eFuses in the destroy path unless you have a specific, tested reason.
- For wear-leveled partitions (NVS, SPIFFS, FAT), erase the ENTIRE partition range rather than deleting keys/files, because the FS layers keep stale copies in spare sectors. One full erase pass is forensically sufficient for NOR flash; skip multi-pass schemes (no magnetic remanence).
- Document the recovery contract explicitly: post-brick the only recovery is host-side esptool write_flash of bootloader+table+app. Verify on a sacrificial board that the bricked unit truly fails to boot (ROM download mode still reachable is fine/expected; that is the re-flash path).
- If maximum anti-forensics is required, combine: flash encryption enabled (ciphertext at rest) + full erase of all partitions + erase of bootloader/table. The erase removes the ciphertext so the un-destroyable key becomes irrelevant. This is the strongest in-firmware posture short of physical die destruction.

## Securely wiping an SD card from an ESP32 (best-effort data destruction)

**Summary.** An ESP32 can perform a best-effort secure wipe of an attached SD card, but it cannot guarantee 100% destruction. SD cards are managed NAND flash with a Flash Translation Layer (FTL): wear-leveling, over-provisioning, and bad-block remapping mean host-issued writes never reach all physical cells, so remapped/retired blocks may retain recoverable data. The honest, achievable goal is: enumerate and overwrite every file, then overwrite the entire LBA space (all sectors / free space) with hardware random data via esp_fill_random, optionally issue the card's flash erase, then reformat. For the wipe loop you want maximum throughput and raw block access, which favors SD_MMC 4-bit (ESP32/S3 only, fixed pins) or SdFat raw-sector writes over the stock SPI SD library. ESP32 Marauder almost always mounts the SD over SPI (sharing the TFT SPI bus, per-board SD_CS), so on Marauder hardware the wipe must use the SD/SdFat SPI path, not SD_MMC. Realistic full-card overwrite times: ~8GB ≈ 0.5–9 h, 32GB ≈ 2–34 h, 128GB ≈ 8–135 h, depending heavily on interface and buffer size (SPI ~0.26–1 MB/s write; SD_MMC 4-bit raw ~4–5 MB/s). Format alone is NOT secure; overwrite-then-format is the correct procedure.

### Findings

**[high] SD_MMC (SDIO) is the fastest interface but is limited to ESP32/ESP32-S3 with fixed pins; SD-over-SPI is universal but ~half to one-third the speed; SdFat adds raw-sector access and exFAT.**

Espressif SD_MMC README: 'SD_MMC is only available for ESP32 and ESP32-S3. For other SoCs please use the SD library based on SPI.' Default 4-bit mode uses fixed pins on ESP32: CLK=GPIO14, CMD=GPIO15, D0=GPIO2, D1=GPIO4, D2=GPIO12, D3=GPIO13 (all need 10k pullups; even unused D3 must be pulled up or the card drops to SPI mode). ESP32-S3 defaults: CLK=36,CMD=35,D0=37,D1=38,D2=33,D3=34, reconfigurable via setPins(). 1-bit mode via SD_MMC.begin(mountpoint, mode1bit=true). README speed note: '1-bit: SD_MMC ~2x faster than SPI; 4-bit: ~3x faster than SPI.' The stock SD library runs over SPI (4 data + 2 power pins, up to 80MHz, any GPIO via matrix). SdFat (greiman/SdFat) adds raw card access and is the only one of the three with built-in exFAT support; it works on ESP32 but is not bundled in arduino-esp32.

Sources: https://github.com/espressif/arduino-esp32/blob/master/libraries/SD_MMC/README.md, https://github.com/espressif/arduino-esp32/blob/master/libraries/SD/README.md, https://github.com/greiman/SdFat, https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/peripherals/sdmmc_host.html

**[high] esp_fill_random() is the correct RNG for the overwrite payload, but it only emits TRUE random when an entropy source is active (Wi-Fi/BT on, or bootloader_random_enable()); otherwise it is pseudo-random.**

Signatures: 'uint32_t esp_random(void)' and 'void esp_fill_random(void *buf, size_t len)' — buf = pointer to buffer to fill, len = length in bytes. esp_fill_random is implemented via esp_random(). For true randomness at least one must hold: (1) RF on via esp_wifi_start()/esp_bt_controller_enable(); (2) bootloader_random_enable() (SAR-ADC entropy) called before use and bootloader_random_disable() after; (3) running in 2nd-stage bootloader. Otherwise output 'should be considered as pseudo-random only.' The HW RNG busy-waits to bound read frequency to ~15–75 kHz, but esp_fill_random buffers efficiently so this does not bottleneck multi-KB fills. For a wipe, pseudo-random is acceptable (the goal is to obliterate, not to seed crypto), but turning Wi-Fi on first makes it true-random for free — and on Marauder Wi-Fi is already available.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/system/random.html

**[high ⚠HIGH-RISK] Overwrite cannot guarantee 100% destruction on flash SD cards — FTL wear-leveling, over-provisioning, and bad-block remapping leave physically retained copies the host cannot address. State this honestly; recommend full-card overwrite + erase + reformat as best-effort.**

NIST SP 800-88 Rev.2 treats flash media (incl. memory cards) like SSDs: standard overwrites may not reach all physical locations because of wear-leveling, over-provisioning, and bad-block management; data can persist in remapped cells, over-provisioned space, and blocks awaiting garbage collection. UCSD's 2011 study showed whole-drive overwrites of SSDs still left substantial recoverable data in some cases. Consequence: an ESP32 (or any host) writing only the logical LBA space cannot touch retired/over-provisioned physical pages. Best-effort posture: (a) overwrite every file, (b) overwrite the entire LBA range / all free space with random data (1 pass is sufficient on flash — multi-pass is an HDD-era ritual that adds wear/time with negligible benefit on flash since you can't reach the hidden cells anyway), (c) optionally issue the card's native flash erase (SdFat card->erase), (d) reformat. For true assurance the only reliable method is physical destruction of the NAND. This limitation is load-bearing for any 'secure wipe' claim in product docs.

Sources: https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-88r2.pdf, https://expungedata.com/blog/nist-800-88-rev2-explained, https://www.drivewipe.com/standards/nist-800-88

**[high] Single-pass random overwrite is the right choice on flash; multi-pass gives no meaningful security gain and roughly multiplies wipe time and wear.**

Because the FTL hides remapped/over-provisioned cells from the host regardless of how many times you rewrite addressable LBAs, additional passes do not recover the unreachable data and do not improve assurance. NIST 800-88 abandoned the old multi-pass overwrite prescriptions for exactly this class of media. Recommendation: 1 random pass over the full card, then format. Reserve any extra effort for the native ERASE command rather than extra overwrite passes.

Sources: https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-88r2.pdf, https://www.drivewipe.com/standards/nist-800-88

**[high] Throughput is the dominant time driver; realistic ESP32 write rates: SPI ~0.26–1 MB/s, SD_MMC 4-bit FATFS ~0.6–4 MB/s, raw SD_MMC sector writes ~4–7 MB/s with 32–64KB buffers.**

Measured: SD-SPI write commonly ~0.26–0.4 MB/s with small buffers (Schatzmann/Arduino-forum/atomic14), reaching ~1 MB/s only with large contiguous writes and fast SPI clock. SD_MMC 4-bit via FATFS measured ~0.59 MB/s write / 1.45 MB/s read in one test, up to ~4 MB/s with large buffers. Raw sdmmc_write_sectors (bypassing FAT): 4.93 MB/s @32KB, 5.59 MB/s @64KB, up to ~7 MB/s sequential. FATFS is far slower than raw because it can't coalesce sectors — so a fast wipe should write large (32–64KB) buffers, ideally to raw sectors. Buffer size is the single biggest tunable: <100B writes cap ~0.4 MB/s.

Sources: https://www.pschatzmann.ch/home/2025/03/25/test/, https://forum.arduino.cc/t/sd-card-benchmark-with-esp32/987413, https://github.com/espressif/esp-idf/issues/11628, https://www.atomic14.com/videos/posts/ocXs1yxsux4

**[high] Full-card overwrite time estimates (single pass): 8GB ≈ 0.5–9h, 32GB ≈ 2–34h, 128GB ≈ 8.5–135h, bracketed by SD_MMC-raw (best) and SPI-slow (worst).**

Time = capacity / write-rate. At raw SD_MMC ~5 MB/s: 8GB≈27min, 32GB≈1.8h, 128GB≈7.1h. At a moderate ~2 MB/s (SD_MMC FATFS / tuned SPI large-buffer): 8GB≈1.1h, 32GB≈4.4h, 128GB≈17.8h. At slow SPI ~0.26 MB/s: 8GB≈8.5h, 32GB≈34h, 128GB≈137h. (1 GB taken as 1000 MB; using GiB raises numbers ~7%.) Practical guidance for Marauder (SPI-bound): tune SPI clock high and use 32–64KB buffers to stay near the ~1 MB/s end (8GB≈2.2h, 32GB≈8.9h, 128GB≈35.6h) rather than the 0.26 MB/s worst case. Always size expectations to the interface the target board actually uses.

Sources: https://www.pschatzmann.ch/home/2025/03/25/test/, https://github.com/espressif/esp-idf/issues/11628

**[high] FAT32 is the wrong target for >32GB cards: it has a 4GB max file size and Windows' formatter caps FAT32 at 32GB; SDXC cards (>32GB) ship as exFAT, which is the SD Association default for >32GB.**

FAT32 individual file limit is 4GB; the Windows GUI restricts FAT32 formatting to <=32GB volumes (the on-disk format itself can go larger; recent Win11 builds raised the GUI cap to 2TB, but cross-device compatibility still expects exFAT on big cards). SDXC (>32GB) and SDUC must support exFAT as default. Practical rule for the wipe-then-reformat step: <=32GB -> FAT32; >32GB -> exFAT. SdFat's SdFormatter example encodes exactly this threshold (see code finding).

Sources: https://en.wikipedia.org/wiki/ExFAT, https://www.integralmemory.com/faq1/what-are-the-differences-between-fat16-fat32-and-exfat-file-systems/, https://it.slashdot.org/story/24/08/16/1440221/microsoft-removes-fat32-partition-size-limit-in-windows-11

**[high] SdFat exposes the exact primitives needed: card->erase(firstSector,lastSector), card->sectorCount(), raw card via SdCardFactory, and FatFormatter/ExFatFormatter — with a built-in 32GB (67,108,864-sector) FAT32-vs-exFAT decision.**

SdCardInterface declares 'virtual bool erase(Sector_t firstSector, Sector_t lastSector) = 0;'. SdFormatter.ino: 'm_card = cardFactory.newCard(SD_CONFIG);' then 'cardSectorCount = m_card->sectorCount();'. Erase loop: 'ERASE_SIZE = 262144L' sectors per call, loop 'm_card->erase(firstBlock,lastBlock); firstBlock += ERASE_SIZE; while (firstBlock < cardSectorCount);'. Format decision: 'cardSectorCount > 67108864 ? exFatFormatter.format(...) : fatFormatter.format(...)' (67,108,864 sectors x 512B = 32GiB). NOTE: SD card 'erase' sets erased data to 0 or 1 per vendor and requires single-block-erase support; treat it as a complement to, not a replacement for, the random overwrite. For the random-overwrite pass, write 512B*N buffers filled by esp_fill_random via writeSectors() over [0, sectorCount).

Sources: https://github.com/greiman/SdFat-beta/blob/master/examples/SdFormatter/SdFormatter.ino, https://files.asec01.net/esp8266/libraries/ESP8266SdFat/extras/html/class_sd2_card.html

**[high ⚠HIGH-RISK] ESP32 Marauder mounts the SD over SPI (typically sharing the TFT SPI bus); SD_CS varies per board and at least one supported board (M5StickC) has NO SD card.**

From Marauder configs.h: most boards define SD_MISO=TFT_MISO, SD_MOSI=TFT_MOSI, SD_SCK=TFT_SCLK (shared SPI bus). SD_CS per board: V4/V6/KIT/Cardputer=12, V6_1=14, Mini=4, V7/V7_1=4, Mini_V3/V8/C5/Flipper/Multiboard_S3=10, all CYD variants=5 (CYD data lines MISO=19/MOSI=23/SCK=18), Rev_Feather=5, XIAO_S3=3, M5StickC SD_CS=-1 (no SD). Implication: on Marauder you cannot use SD_MMC 4-bit (pins are used by TFT and the SD is wired for SPI); the wipe must run on the SD/SdFat SPI path using that board's SD_CS, and expect SPI-class throughput. Also note the CYD VSPI contention (SD vs touchscreen share VSPI) — wipe with the display/touch idle.

Sources: https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h, https://randomnerdtutorials.com/esp32-cyd-display-touchscreen-microsd-card/, https://github.com/justcallmekoko/ESP32Marauder/wiki/cyd

**[high] Format alone is NOT secure; the correct procedure is overwrite-then-format.**

A FAT/exFAT format only rewrites filesystem metadata (FATs, root dir, a few system sectors); the bulk of data sectors are untouched and trivially recoverable with carving tools. Secure-by-best-effort order: (1) recursively enumerate and overwrite each file's contents (defeats casual recovery and zeroes obvious metadata), (2) overwrite the entire LBA space / all free space with esp_fill_random data (single pass), (3) optionally card->erase() the full range, (4) reformat to FAT32 (<=32GB) or exFAT (>32GB) so the card is usable again. Document that even this is best-effort due to FTL-hidden cells; physical destruction is the only guarantee.

Sources: https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-88r2.pdf, https://github.com/greiman/SdFat-beta/blob/master/examples/SdFormatter/SdFormatter.ino

### Recommendations
- Adopt this best-effort wipe order and document it as best-effort (not guaranteed): (1) recursive file overwrite+delete, (2) single-pass random overwrite of the FULL LBA range via esp_fill_random, (3) optional native card->erase() over [0,sectorCount), (4) reformat (FAT32 if <=32GB, exFAT if >32GB). Always state the FTL limitation in user-facing docs.
- Single pass only on flash. Do not implement multi-pass (Gutmann/DoD-style) — it multiplies time/wear with no security gain because the FTL hides remapped/over-provisioned cells from any host write. Spend the extra effort on the native ERASE command instead.
- Choose the interface per the target board. On ESP32/ESP32-S3 standalone, prefer SD_MMC 4-bit (fixed pins, 10k pullups incl. D3) or SdFat raw-sector writes for ~5 MB/s. On ESP32 Marauder, you are SPI-bound: use the SD/SdFat SPI path with that board's SD_CS (e.g. CYD=5, V7=4, V8/Mini_V3/Flipper=10) and skip M5StickC (SD_CS=-1, no card).
- Maximize throughput: write 32–64KB buffers (multiples of 512B), push SPI clock as high as the card allows (SD_SCK_MHZ(40–50)), and prefer raw writeSectors() over FATFS file writes — FATFS can't coalesce and is several times slower. Refill the buffer with esp_fill_random once per buffer, not per sector.
- Enable an entropy source before wiping so esp_fill_random is TRUE random: simplest is esp_wifi start (already on for Marauder) or bootloader_random_enable() at start / bootloader_random_disable() at end. Pseudo-random is acceptable for destruction, but true-random is nearly free here.
- Show progress and ETA from sectorCount and measured MB/s, and warn the user up front of realistic durations (SPI 32GB can be 9–34h; SD_MMC-raw 32GB ~2h). Make the operation resumable/abortable and require an explicit confirm — this is irreversible.
- For provable destruction (high-assurance threat models), state plainly that software wipe is best-effort and recommend physical destruction of the card as the only guarantee per NIST 800-88 'Destroy'.

### Open questions
- Which exact Marauder board(s) is this targeting? It determines SD_CS, whether an SD exists at all (M5StickC has none), and the achievable throughput / ETA shown to the user.
- Does the wipe need to run from within Marauder firmware (constrained to its SD/SPI mount and shared TFT bus) or from a dedicated standalone sketch where SD_MMC 4-bit / SdFat raw is allowed for much faster wipes?
- Should the implementation be Arduino (SD/SdFat/SD_MMC libraries) or ESP-IDF (esp_vfs_fat + sdmmc/sdspi + raw sdmmc_write_sectors)? IDF gives cleaner raw-sector access and explicit bootloader_random control.
- Is reformat-to-usable required after wiping, or is leaving the card raw/blank acceptable? This affects whether to bundle FatFormatter/ExFatFormatter.
- What is the threat model / assurance bar? If it must defeat lab forensics, software wipe is insufficient and the design should require physical destruction — this changes the product claims substantially.
- Will the card sizes in scope exceed 32GB (SDXC/exFAT) in practice, and is the host ecosystem OK with exFAT, or must everything come back as FAT32 (<=32GB only)?

## ESP32 hardware arming / dead-man switch read at boot for a self-destruct (secure-wipe) trigger — strapping-pin avoidance, safe GPIO selection, fail-safe pull-resistor design, and per-board free-pin availability (classic ESP32, S3, C3; headless dev boards, CYD, Marauder Mini, M5Cardputer, M5StickC)

**Summary.** An arming switch read at boot must NOT live on a strapping/boot pin, because the same logic level you read for "armed" can silently divert the chip into UART/USB download mode, lock the part in 1.8V flash mode and brown out, or be overridden by the chip's own internal pull at reset — all of which defeat the security goal. On classic ESP32 avoid GPIO0/2/5/12(MTDI)/15(MTDO), the flash pins GPIO6-11, and remember GPIO34-39 are input-only with NO internal pull resistors. On ESP32-S3 avoid GPIO0/3/45/46 (and the SPI-flash/PSRAM pins ~26-37); on ESP32-C3 avoid GPIO2/8/9 (and flash pins ~12-17). For the threat model (tamper/wipe), the correct fail-safe default is "absence of the armed signal = DISARMED = wipe": use INPUT_PULLDOWN (or an external pull-down) so a cut, unplugged, or floating wire reads LOW, and wire the switch to drive the pin HIGH (to 3.3V) only when physically in the armed position. This makes removing the switch, cutting the harness, or de-soldering the magnet all trigger the wipe. Read the pin in early boot AFTER a short settle/debounce, sample multiple times, and treat any non-confirmed-armed result as disarmed. On display/keyboard boards (CYD, Marauder, Cardputer, StickC) almost every GPIO is consumed; the few truly-free pins (e.g. CYD GPIO22/27, Cardputer/StickC Grove G1/G2) are the only realistic landing spots.

### Findings

**[high ⚠HIGH-RISK] Classic ESP32 strapping/boot pins to AVOID for an arming switch are GPIO0, GPIO2, GPIO5, GPIO12 (MTDI), GPIO15 (MTDO). GPIO0 LOW at reset = UART download/bootloader; GPIO2 must be floating/LOW; GPIO5 and GPIO15 default HIGH.**

GPIO0 has an internal pull-up and selects download mode when held LOW at reset (so an arming switch that pulls GPIO0 low would drop the device into the flasher instead of booting your firmware). GPIO2 has effectively a pull-down/floating default and must be LOW or floating to enter download mode; it is 'the most common cause of boot problems.' GPIO15(MTDO) defaults HIGH and silences ROM boot messages when LOW; GPIO5 defaults HIGH (SDIO timing). Any external switch forcing these levels at reset can change boot behavior. Espressif's own note: GPIO0 etc. have a ~45k internal pull, so a switch must use a STRONG pull (e.g. 10k to rail) to override it — exactly the kind of fragile arrangement to avoid on a security input.

Sources: https://www.espboards.dev/blog/esp32-strapping-pins/, https://docs.espressif.com/projects/esptool/en/latest/esp32/advanced-topics/boot-mode-selection.html, https://randomnerdtutorials.com/esp32-pinout-reference-gpios/

**[high ⚠HIGH-RISK] GPIO12 (MTDI) is the single most dangerous strapping pin: HIGH at boot forces VDD_SDIO flash voltage to 1.8V, which can brown out / damage 3.3V-flash modules and prevent boot or flashing entirely. NEVER put an arming switch on GPIO12.**

GPIO12 has an internal pull-DOWN and must be LOW at reset on standard 3.3V-flash modules. If an external switch (or a long floating wire that couples noise) pulls GPIO12 high at reset, the flash regulator is set to 1.8V and the module will fail to boot or be damaged. Espressif/esptool: pulling GPIO12 high with a 3.3V flash 'May prevent flashing and/or booting ... causing the flash to brownout.' This is doubly disqualifying for a security pin because a tamper attempt that grounds/floats the harness could brick the device in a way that does NOT trigger your wipe.

Sources: https://docs.espressif.com/projects/esptool/en/latest/esp32/advanced-topics/boot-mode-selection.html, https://www.espboards.dev/blog/esp32-strapping-pins/

**[high ⚠HIGH-RISK] Classic ESP32 GPIO34, 35, 36(VP), 39(VN) are INPUT-ONLY and have NO internal pull-up or pull-down resistors — they require an external resistor and must not be relied on for a fail-safe floating-default read.**

RandomNerdTutorials (corroborated by multiple pinout refs): 'GPIOs 34 to 39 are GPIs – input only pins. These pins don't have internal pull-up or pull-down resistors.' Consequence for the design: you CAN use 34-39 to READ the switch, but you MUST add an external pull resistor (10k) yourself, because INPUT_PULLDOWN/INPUT_PULLUP will silently do nothing on these pins, leaving a floating, indeterminate read at boot. Also note GPIO6-11 are wired to the SPI flash and must never be touched.

Sources: https://randomnerdtutorials.com/esp32-pinout-reference-gpios/, https://lastminuteengineers.com/esp32-pinout-reference/

**[high ⚠HIGH-RISK] ESP32-S3 strapping/boot pins to AVOID are GPIO0, GPIO3, GPIO45, GPIO46. GPIO0 (pull-up) selects download when LOW; GPIO46 (pull-down) must be floating/LOW for bootloader; GPIO45 sets VDD_SPI flash voltage (the S3 analog of classic GPIO12); GPIO3 controls JTAG source and has NO internal pull (must be set externally).**

Per esptool S3 boot-mode doc + datasheet strapping section: GPIO0 LOW = ROM serial bootloader, HIGH = normal SPI flash boot, internal pull-up so floats HIGH. GPIO46 'must be either left unconnected/floating, or driven Low, in order to enter the serial bootloader' and is ignored when GPIO0 is high. GPIO45 controls VDD_SPI 1.8V/3.3V (avoid like GPIO12). GPIO3 has no internal pull resistor (JTAG select) so it floats unpredictably — bad for a security input. Additionally, on S3 modules with in-package or external flash/PSRAM the SPI pins (roughly GPIO26-32, and 33-37 on octal-PSRAM WROOM-1 N16R8) are consumed and must be avoided.

Sources: https://docs.espressif.com/projects/esptool/en/latest/esp32s3/advanced-topics/boot-mode-selection.html, https://www.espboards.dev/blog/esp32-strapping-pins/, https://makerspet.com/blog/esp32-s3-gpio-limitations/

**[high ⚠HIGH-RISK] ESP32-C3 strapping/boot pins to AVOID are GPIO2, GPIO8, GPIO9. GPIO9 has an internal pull-UP (the BOOT/Flash button pulls it LOW for download). GPIO8 must be HIGH to enter the bootloader reliably, and the combination GPIO8=0 AND GPIO9=0 is INVALID and causes undefined behavior.**

Espressif hardware-design-guidelines reset table: GPIO9 reset state = 'input enabled, pull-up enabled (IE=1, WPU=1)', so it floats HIGH (normal SPI boot); pulling it LOW at reset = Joint Download Boot. GPIO2 and GPIO8 default to input/high-Z (no internal pull noted). 'GPIO8 = 0 and GPIO9 = 0 is invalid and will trigger unexpected behavior.' Espressif also warns: 'Do not add high-value capacitors at GPIO9, or the chip may enter download mode' — i.e. switch/wire capacitance on GPIO9 can falsely trigger the flasher. Avoid these three for the arming switch. C3 flash pins (~GPIO12-17) are also reserved.

Sources: https://docs.espressif.com/projects/esptool/en/latest/esp32c3/advanced-topics/boot-mode-selection.html, https://docs.espressif.com/projects/esp-hardware-design-guidelines/en/latest/esp32c3/schematic-checklist.html, https://www.espboards.dev/blog/esp32-strapping-pins/

**[high] Recommended SAFE free GPIOs for an arming switch (general/headless boards): classic ESP32 — GPIO4, 13, 16, 17, 18, 19, 21, 22, 23, 25, 26, 27, 32, 33 (all support internal pull-up AND pull-down). ESP32-S3 — GPIO4-18 and ~GPIO38-42 (avoid 0/3/45/46 and the flash/PSRAM range). ESP32-C3 — GPIO3, 4, 5, 6, 7, 10 (avoid 2/8/9, 11=VDD_SPI, and 12-17 flash).**

On the classic ESP32 these GPIOs have full bidirectional pull-up/pull-down support and are not sampled at reset, so they are immune to the boot-mode and 1.8V-flash hazards. Prefer an RTC-capable, non-strapping pin so it is also usable for deep-sleep wake if you later want the tamper line to wake-and-wipe (e.g. GPIO32/33 on classic, GPIO0-7 RTC domain on S3/C3 — but skip the strapping ones). The article principle: 'Prioritize non-strapping pins for critical hardware. Save strapping pins for non-essential features.'

Sources: https://randomnerdtutorials.com/esp32-pinout-reference-gpios/, https://www.espboards.dev/blog/esp32-strapping-pins/, https://docs.espressif.com/projects/arduino-esp32/en/latest/api/gpio.html

**[high ⚠HIGH-RISK] FAIL-SAFE DESIGN (load-bearing): the safe default for a tamper/dead-man arming line is 'no confirmed-armed signal = DISARMED = wipe.' Use INPUT_PULLDOWN (or external 10k pull-down) and make the switch drive the pin HIGH to 3.3V only in the armed position. A cut, unplugged, corroded, or floating wire then reads LOW = disarmed = triggers wipe.**

Reasoning: a true anti-forensic / dead-man design must FAIL TOWARD the destructive action, not away from it. If you instead used INPUT_PULLUP with armed=LOW (the conventional button wiring), then cutting the wire or removing the switch reads HIGH = 'armed/safe' = NO wipe — an attacker just snips the harness and the device boots normally. Inverting to pull-DOWN + active-HIGH-armed means every tamper vector (cut harness, removed magnet on a reed/hall case-open switch, desoldered switch, water ingress shorting to ground) collapses the pin to LOW = disarmed = wipe. The internal/external pull resistor is what enforces the default: it defines the value the pin holds when the switch is absent, so it must point at the SAFE-to-destroy state. Caveat: a determined attacker who instead shorts the pin to 3.3V could hold it 'armed'; to resist that, combine with a second criterion (e.g. signed token, expected resistance via ADC voltage-divider 'closed-loop' so both a short to GND and a short to VCC read as wrong = wipe).

Sources: https://docs.espressif.com/projects/arduino-esp32/en/latest/api/gpio.html, https://randomnerdtutorials.com/esp32-pinout-reference-gpios/, https://microcontrollerslab.com/push-button-esp32-gpio-digital-input/

**[high] INPUT_PULLUP / INPUT_PULLDOWN config: ESP32 internal pulls are ~45k. INPUT_PULLUP ties the pin to 3.3V through the internal resistor; INPUT_PULLDOWN ties it to GND. If neither is set the pin floats (high-impedance) and reads garbage — unacceptable for a boot-time security read. Input-only pins 34-39 ignore both and need external resistors.**

Arduino-ESP32 GPIO API: 'The ESP32 SoC families support internal pullup and pulldown through a 45k resistor... enabled when configuring the GPIO mode as INPUT.' Use pinMode(PIN, INPUT_PULLDOWN) for the active-HIGH-armed fail-safe scheme. For an SPDT toggle, wire the common to the GPIO, one throw to 3.3V (armed) and optionally the other throw to GND (explicit disarm) — but the pull resistor must still define the floating state. For a reed / tilt / magnetic case-open switch (typically a simple normally-open or normally-closed contact), choose the contact orientation so that the INTACT, SEALED, ARMED state is the one driving the pin to the armed level, and the disturbed state releases to the pull-resistor default.

Sources: https://docs.espressif.com/projects/arduino-esp32/en/latest/api/gpio.html, https://microcontrollerslab.com/push-button-esp32-gpio-digital-input/

**[high] Reading reliably AT BOOT: read the arming pin in early app startup (not in the bootloader window), after a short settle delay, and DEBOUNCE by sampling multiple times. Mechanical/reed switches bounce; typical debounce 5-20ms, but large reed switches can bounce up to ~100ms.**

Practical recipe: pinMode(ARM_PIN, INPUT_PULLDOWN); delay a few ms for the line/pull to settle; sample the pin N times over ~20-50ms (or integrate over 100ms for a reed) and require a stable, unanimous 'armed' reading to consider the device armed; ANY ambiguity or a single disarmed read => treat as disarmed. Because the pin is NOT a strapping pin, you avoid the reset-window sampling problem entirely and can take your time in setup(). Sources: ESPHome/SwitchDoc note reed switches 'may bounce up to about 100ms'; general ESP32 debounce 5-20ms. For a wake-from-deep-sleep tamper line, use an RTC-capable safe GPIO with ext0/ext1 wake.

Sources: https://www.switchdoc.com/2018/04/esp32-tutorial-debouncing-a-button-press-using-interrupts/, https://community.home-assistant.io/t/esp32-binary-sensor-bouncing/373967, https://docs.espressif.com/projects/arduino-esp32/en/latest/api/gpio.html

### Recommendations
- Default pin + level: use GPIO27 on classic ESP32 (and CYD), or Grove G2 on ESP32-S3 boards like Cardputer; configure INPUT_PULLDOWN; ARMED = pin driven HIGH (3.3V) by the switch. Any LOW/floating/cut reading = DISARMED = trigger wipe.
- Fail-safe rule (most important): wire so the INTACT, SEALED, ARMED state is the only thing that drives the pin to the armed level. Use a pull-DOWN as the default so a cut/unplugged/floating/removed switch reads LOW = wipe. Do NOT use the conventional INPUT_PULLUP + active-LOW button wiring, because cutting the wire would read 'safe' and defeat the dead-man property.
- Never place the arming switch on a strapping pin: classic ESP32 GPIO0/2/5/12/15; S3 GPIO0/3/45/46; C3 GPIO2/8/9. Especially never GPIO12(MTDI)/S3 GPIO45 (1.8V flash brownout) or GPIO0/GPIO9 (drops into UART/USB download mode).
- Never use flash/PSRAM pins (classic GPIO6-11; S3 ~26-37 on octal parts; C3 12-17) and remember classic GPIO34-39 (and CYD GPIO35, StickC G36) are INPUT-ONLY with no internal pull — they need an EXTERNAL 10k pull-down to work in this scheme.
- Read at boot in setup() (not in the reset/strapping window): set INPUT_PULLDOWN, settle ~10ms, sample 8+ times over 40ms (100ms for a reed/hall), require unanimous ARMED; treat any ambiguity as disarmed. This avoids strapping timing issues entirely since the pin isn't a boot pin.
- Harden against the obvious bypass (attacker jumpers the pin to 3.3V): use a resistor-coded loop on an ADC pin so the 'armed' state is a specific voltage WINDOW; both short-to-GND and short-to-VCC read out-of-window = wipe. Optionally pair with a signed/eFuse token so a single pin short can't authorize boot.
- If you want tamper-while-asleep to wipe, pick an RTC-capable safe GPIO (classic GPIO32/33) and use ext0/ext1 deep-sleep wake on the disarm transition, then wipe on wake.
- Per-board reality check before committing: CYD -> GPIO27 (CN1); Cardputer/M5 S3 -> Grove G2/G1; M5StickC Plus2 -> G32 or G33; ESP32 Marauder/Mini -> verify your exact variant's schematic (pin budget is nearly exhausted; GPIO12 is even used as SD_CS) and pick a confirmed-free non-strapping pin like GPIO25/26 only after checking.
- Add a series resistor (~1k) and a small (but NOT large) RC only on the safe GPIO if you need ESD/noise protection; avoid large capacitors on any strapping-adjacent line (Espressif warns C3 GPIO9 caps can force download mode).

### Open questions
- The exact free-pin map for the specific 'Marauder Mini' variant in your build is not nailed down — Marauder hardware varies (ESP8266 companion, CC1101/NRF/SD options). Confirm against your board's actual schematic which non-strapping GPIO is truly unused before wiring the arming switch.
- Threat model decision: do you want to also resist an attacker who can OPEN the case and short the arming pin to 3.3V? If yes, adopt the ADC resistor-window / signed-token hardening; if the threat is only 'someone powers it on or cuts the harness,' the simple pull-down + active-HIGH scheme suffices.

### Adversarial verifications
- **[CONFIRMED]** Classic ESP32 strapping/boot pins to AVOID for an arming switch are GPIO0, GPIO2, GPIO5, GPIO12 (MTDI), GPIO15 (MTDO). GPIO0 LOW at reset = UART download/bootloader; GPIO2 must be floating/LOW; GPIO5 and GPIO15 default HIGH.
  - Verified against two Espressif primary sources: (1) the official ESP32 Series Datasheet v5.2, Section 3 "Boot Configurations" (downloaded from documentation.espressif.com and text-extracted locally), and (2) the official esptool docs "Boot Mode Selection - ESP32".

PIN-BY-PIN CONFIRMATION (Datasheet Table 3-1 "Default Configuration of Strapping Pins"):
- GPIO0: Pull-up, default bit value 1 (HIGH). CONFIRMED.
- GPIO2: Pull-down, default bit value 0 (LOW). CONFIRMED ("pull-down/floating default... must be LOW or floating").
- MTDI/GPIO12: Pull-down, default 0. CONFIRMED.
- MTDO/GPIO15: Pull-up, default bit value 1 (HIGH). CONFIRMED ("defaults HIGH").
- GPIO5: Pull-up, default bit value 1 (HIGH). CONFIRMED ("defaults HIGH").

BOOT BEHAVIOR (Datasheet Table 3-3 "Chip Boot Mode Control"): GPIO0=1 -> SPI Boot Mode (default, runs firmware from flash); GPIO0=0 AND GPIO2=0 -> Joint Download Boot Mode (which the datasheet states supports UART Download Boot and SDIO Download Boot). So GPIO0 held LOW at reset drops into download/bootloader. CONFIRMED. esptool doc reinforces: "The ESP32 will enter the serial bootloader when GPIO0 is held low on reset. Otherwise it will run the program in flash" and "GPIO2 must also be either left unconnected/floating, or driven Low, in order to enter the serial bootloader."

GPIO15/MTDO ROM-message silencing (Datasheet Table 3-4 "U0TXD Printing Control"): MTDO=1 -> Printing Enabled (default); MTDO=0 -> Disabled. So MTDO/GPIO15 driven LOW silences boot messages. CONFIRMED. esptool doc: "If driven Low, silences boot messages printed by the ROM bootloader."

GPIO5 = SDIO timing role (Datasheet Table 3-5 "Timing Control of SDIO Slave"): MTDO and GPIO5 control SDIO slave sampling/output edge timing. CONFIRMED that GPIO5's documented strapping function is SDIO timing.

INTERNAL ~45k PULL / NEED FOR STRONG EXTERNAL PULL: esptool doc states "The ESP32 has a 45k ohm internal pull-up/pull-down resistor at GPIO0 (and other pins)" and that to override boot strapping "this has to be a strong pull-down. For example a 10k resistor to GND." CONFIRMED, and this directly supports the design conclusion that such pins are fragile/contended inputs unsuitable for a security arming switch.

DESIGN CONCLUSION (avoid these pins for an arming switch): Datasheet Section 3 confirms these are latched at reset and that external pull resistors are precisely how their bit values get changed at boot -- so any switch forcing a level on them at reset can alter boot mode. The "avoid GPIO0/2/5/12/15" recommendation is sound.

MINOR INACCURACIES / NUANCES (do not change the verdict, but worth flagging):
1. Strapping-pin COUNT: The datasheet (v5.2) front matter says "Five strapping GPIOs" and Section 3 enumerates exactly GPIO0, GPIO2, MTDI(GPIO12), MTDO(GPIO15), GPIO5. The claim's set of five matches exactly. However, some secondary sources (and older esptool docs) also list GPIO4 as a strapping pin; the official v5.2 datasheet does NOT list GPIO4 as a strapping pin. The claim's five-pin set is correct per the current datasheet.
2. "GPIO2 is the most common cause of boot problems": This phrasing is NOT found in the Espressif primary sources. The esptool doc does not characterize GPIO2 as 'the most common cause.' It is a widely repeated community statement (and plausible, since dev boards often hang LEDs on GPIO2), but it is a community claim, not an Espressif-documented fact. Treat that one sub-clause as folklore-grade, not primary-sourced.
3. GPIO12/MTDI risk: The datasheet confirms MTDI=1 selects 1.8V VDD_SDIO instead of default 3.3V; pulling it HIGH at boot can brown out / damage 3.3V flash modules. This reinforces avoiding GPIO12, consistent with the claim listing it.

Net: every concrete, falsifiable technical assertion in the claim (pin set, default levels, GPIO0 download behavior, GPIO2 download requirement, MTDO message-silencing, GPIO5 SDIO role, 45k internal pull, 10k strong-pull requirement) is corroborated by Espressif primary sources. Only the unsourced 'most common cause of boot problems' editorializing is not primary-sourced.
  - sources: ESP32 Series Datasheet v5.2, Section 3 'Boot Configurations' (Table 3-1 Default Configuration of Strapping Pins; Table 3-3 Chip Boot Mode Control; Table 3-4 U0TXD Printing Control; Table 3-5 Timing Control of SDIO Slave) -- https://documentation.espressif.com/esp32_datasheet_en.pdf (redirected from https://www.espressif.com/sites/default/files/documentation/esp32_datasheet_en.pdf), pages 22-25, text extracted locally via PyMuPDF, esptool docs, 'Boot Mode Selection - ESP32' -- https://docs.espressif.com/projects/esptool/en/latest/esp32/advanced-topics/boot-mode-selection.html (confirms GPIO0 serial-bootloader-when-low + internal pull-up; GPIO2 floating/low requirement; MTDO/GPIO15 low silences ROM boot messages; 45k internal pull and 10k strong external pull to override)
- **[CONFIRMED]** GPIO12 (MTDI) is the single most dangerous strapping pin: HIGH at boot forces VDD_SDIO flash voltage to 1.8V, which can brown out / damage 3.3V-flash modules and prevent boot or flashing entirely. NEVER put an arming switch on GPIO12.
  - The functional core of the claim is directly confirmed by multiple Espressif primary sources. The esptool "Boot Mode Selection" documentation states verbatim that GPIO12/MTDI "Has internal pull-down, so unconnected = Low = 3.3V," that "If driven High, flash voltage (VDD_SDIO) is 1.8V not default 3.3V," and gives the exact warning: "May prevent flashing and/or booting if 3.3V flash is used and this pin is pulled high, causing the flash to brownout." The supporting-detail quote in the claim matches this source almost word-for-word. The ESP-IDF "SD Pull-up Requirements" page independently corroborates the mechanism: "MTDI (GPIO12) is used as a bootstrapping pin to select the output voltage of an internal regulator (VDD_SDIO) which powers the flash chip. This pin has an internal pull-down, so, if left unconnected, it will read low at startup," and that 3.3V-flash modules must have GPIO12 LOW at reset (1.8V-flash WROVER modules instead have an internal pull-up to hold it HIGH). The espefuse "Set Flash Voltage" page confirms the regulator table (MTDI Low/unconnected = 3.3V, High = 1.8V). The engineering conclusion -- never put an arming/tamper switch on GPIO12 -- is therefore well-founded: an external switch or noisy floating harness that pulls GPIO12 high at reset will set 1.8V on a 3.3V flash part and the device will fail to boot/flash (browning out), which for an anti-forensic design is a fail-silent brick that would NOT trigger a wipe routine that never gets to run. The reasoning that this is disqualifying for a security pin is sound. ONE NUANCE worth flagging, which is why I note it rather than rate higher: Espressif's primary docs describe a BROWNOUT / functional failure (fails to boot or flash), NOT permanent physical DAMAGE.   - ESP-IDF docs confirm all parts. 1920K two-OTA app slots are standard for 4MB flash, leaving about 400K free for a 1.5MB bin, consistent with the 409K free figure. NVS, SPIFFS, and SD are separate data partitions storing plaintext or unencrypted data, so the wipe must cover all three. Each ESP32 app partition is a complete self-contained image with no cross-partition code sharing at runtime, so a separate firmware B must compile its own Display, Touch, Keyboard, Switches, and SD drivers, while A inherits drivers from the stock codebase it builds into. Caveat: logical nvs_flash_erase or SPIFFS format does not securely erase flash due to wear-leveling, so the design must use full-region physical erase to be truly anti-forensic.
  - sources: https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-guides/partition-tables.html, https://docs.espressif.com/projects/esp-idf/en/stable/esp32/api-reference/storage/nvs_flash.html, https://developer.espressif.com/blog/switch-between-firmware-binaries/

## Duress / panic / anti-forensic-wipe / dead-man-switch systems — design patterns and safety/failure-mode requirements for an owner-only, single-device data-protection feature (ESP32 boot-gate context)

**Summary.** Across every established system (Kali LUKS Nuke, GrapheneOS duress PIN, BusKill, USBKill, Tails, Purism hardware switches, VeraCrypt hidden volumes), the dominant and converged mechanism is crypto-erase: destroy a small key/header rather than overwrite bulk data, which is fast (BusKill measured ~5 sec to power-off), permanent, and is the modern secure-sanitization standard formalized in NIST SP 800-88 Rev.2. The single most important and consistent design lesson is fear of accidental destruction: BusKill ships only non-destructive triggers by default and deliberately adds friction to destructive ones; GrapheneOS is repeatedly criticized precisely because it has NO confirmation/no-undo path; USBKill has a documented boot-time whitelist-poisoning flaw. The safest designs separate arming (explicit user intent) from triggering (the event), default to the least-destructive action (lock/shutdown, not wipe), and require an out-of-band recovery path — the key insight from Kali Nuke is that you back up the LUKS header (luksHeaderBackup) and store it offline so the owner — and only the owner — can restore. A SAFE/TEST mode that exercises the entire trigger chain while substituting a dummy key for the real one is essential to build confidence without risking real data. Finally, the framing must be explicitly owner-only, defensive, single-device protection against theft/coercion — NOT a tool for evading lawful process, since intentional evidence destruction is obstruction of justice in many jurisdictions and carries serious penalties.

### Findings

**[high ⚠HIGH-RISK] Crypto-erase (destroy the key/header, not the data) is the universal mechanism and the formal modern secure-erase standard.**

Kali LUKS Nuke runs `cryptsetup luksErase` to overwrite the keyslots area, rendering all encrypted data irrecoverable without touching the bulk ciphertext. BusKill's self-destruct shreds only the LUKS header (keyslots + JSON/binary metadata) and measured ~5.16s to full power-off, intentionally prioritizing cutting power to RAM over writing >1GB of random data. GrapheneOS duress erases encryption keys (and the eSIM partition). NIST SP 800-88 Rev.2 Section 3.2 defines Cryptographic Erase (CE) as sanitizing/removing access to the encryption keys, giving high assurance far faster than overwrite — valid ONLY when all sensitive data was encrypted (plaintext needs other methods), with algorithms at >=128-bit strength. Architecture implication for an ESP32 boot-gate: gate access by holding/wrapping the disk/volume key; 'wipe' = destroy that small key material, never attempt bulk overwrite.

Sources: https://www.kali.org/blog/nuke-kali-linux-luks/, https://www.buskill.in/luks-self-destruct/, https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-88r2.pdf, https://grapheneos.org/features, https://www.bitraser.com/blog/cryptographic-erase-and-supported-devices/

**[high ⚠HIGH-RISK] FAIL-SAFE DEFAULT = least-destructive action. Destructive wipe must be opt-in, gated behind deliberate friction, and never the shipped default.**

BusKill ships ONLY non-destructive triggers (lock/shutdown) by default; destructive triggers are not in the app and the developers 'intentionally make it difficult to use destructive triggers' because 'most people probably don't actually want an accidental false-positive to destroy all their data.' Docs route users to external links rather than inline instructions to add friction. USBKill defaults to shutdown, not wipe. Default cascade for an ESP32 design should be: lock-only -> shutdown -> (separately, explicitly armed) crypto-erase. The destructive tier should require a distinct, documented opt-in (e.g., a config flag plus a physical jumper/build-time choice), with emphatic warnings.

Sources: https://docs.buskill.in/buskill-app/en/dev/software_usr/destructive.html, https://www.buskill.in/luks-self-destruct/, https://github.com/hephaest0s/usbkill

**[high ⚠HIGH-RISK] Lack of a confirmation/undo path is the #1 documented failure mode — accidental or mistaken entry wipes irreversibly.**

GrapheneOS duress is widely criticized: 'A duress PIN you forget, confuse with your normal PIN, or enter by accident wipes your phone just as effectively as in a genuine emergency... no confirmation screen and no second chance.' This is a deliberate trade-off (the wipe must be uninterruptible to be useful under coercion), but it means the design MUST compensate elsewhere: (1) make the duress credential structurally distinct/hard to hit by accident (recommended mitigation: different LENGTH from the normal PIN, e.g. 6-digit normal vs 4-digit duress), (2) ensure the real/legitimate credential ALWAYS takes precedence so a collision can never trigger a wipe (GrapheneOS: if duress == real PIN, real unlock wins, no wipe), (3) require explicit ARMING so the destructive path is inert until the owner opts in.

Sources: https://www.rebeltech.co.nz/grapheneos-duress-password/, https://www.androidauthority.com/grapheneos-duress-pin-3584795/, https://discuss.grapheneos.org/d/13213-duress-pin-thoughts

**[high ⚠HIGH-RISK] A recovery/provisioning safeguard (owner-only out-of-band backup) is essential and is the established pattern — the wipe must be recoverable by the owner, not by an attacker.**

Kali Nuke's documented workflow: BEFORE relying on nuke, run `cryptsetup luksHeaderBackup --header-backup-file luksheader.back /dev/sda5`, encrypt that backup separately (OpenSSL), and store it offline (USB in a safe, ideally outside a surveillance jurisdiction). After a nuke, recover via `cryptsetup luksHeaderRestore --header-backup-file luksheader.back /dev/sda5` then boot with the original passphrase. BusKill confirms the same: 'the LUKS header could be restored after self-destruct if a backup were made.' Key property: the recovery secret lives OFF the device, so seizing the device alone yields nothing, but the owner can rebuild. ESP32 implication: provisioning must produce an encrypted, off-device escrow of the key/header that only the owner controls; never store the recovery secret on the protected device.

Sources: https://www.kali.org/blog/nuke-kali-linux-luks/, https://www.buskill.in/luks-self-destruct/

**[high] An explicit SAFE/TEST mode that simulates the full trigger chain WITHOUT destroying real data is a derived best practice — and a gap in most shipped systems.**

BusKill's guidance is to 'limit yourself to just the lock screen trigger' for the first few days to learn the disarm habit before enabling destructive behavior, and to test destructive scripts only on 'a disposable machine containing no valuable data.' This is a workaround for the absence of a built-in dry-run. The strong requirement for a new design: a TEST/SAFE mode that runs the entire detection -> arming-check -> trigger -> 'erase' pipeline against a DUMMY key (or logs the action) so the owner can validate wiring, timing, and the trigger condition with zero risk to the real key. Distinguish clearly in the UI/logs between SAFE (simulated) and ARMED/LIVE (will destroy).

Sources: https://docs.buskill.in/buskill-app/en/stable/faq.html, https://www.buskill.in/luks-self-destruct/, https://docs.buskill.in/buskill-app/en/dev/software_usr/destructive.html

**[high ⚠HIGH-RISK] Arming/disarming must be explicit, easy to toggle, and the system must default to DISARMED — separate 'intent to protect' from the trigger event.**

BusKill is armed/disarmed from CLI or GUI and supports temporary disarm (pause) via keyboard shortcut for bathroom breaks etc.; guidance is to disarm before moving the laptop. This separation prevents normal activity from causing destruction. For an ESP32 boot-gate the arming state itself needs a deliberate fail-safe default: choose DISARMED at power-on/after-provisioning so an un-configured or freshly flashed device can never wipe. Make arming state observable (LED/indicator) and require a positive action to enter the LIVE/destructive tier. Note the tension with dead-man-switch semantics (where absence-of-presence is the trigger): resolve by requiring BOTH (a) explicit arming AND (b) the trigger condition, so neither alone destroys data.

Sources: https://docs.buskill.in/buskill-app/en/stable/faq.html, https://github.com/buskill/buskill-app

**[high ⚠HIGH-RISK] A wrong-attempt counter and rate-limiting/lockout interact with the wipe trigger and need carefully chosen fail-safe defaults to avoid both false wipes and brute-force bypass.**

GrapheneOS: failed duress/normal entries still count toward the standard attempt limit and repeated wrong attempts lock the device — i.e., the attempt counter and the duress path coexist. Design tension: a 'wipe after N wrong attempts' counter is itself an accidental-destruction hazard (a child/pocket/curious bystander could trip it), so N must not be too small and the destructive consequence should be gated behind ARMED state. Conversely, rate-limiting/lockout (exponential backoff after wrong tries) is the fail-safe that defends the KEY against brute force without destroying data — prefer lockout/backoff as the default response to wrong attempts, and reserve actual crypto-erase for the explicit duress credential or an armed N-strikes mode the owner knowingly enabled. Persist the counter across reboots so power-cycling can't reset it (counter must be tamper-resistant/monotonic on the ESP32).

Sources: https://www.rebeltech.co.nz/grapheneos-duress-password/, https://www.androidauthority.com/grapheneos-duress-pin-3584795/, https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-88r2.pdf

**[high] Trigger-source robustness matters: physical/dead-man triggers can false-positive, and software whitelists can be poisoned at boot.**

BusKill uses a magnetic breakaway tuned 'strong enough that it won't come undone unexpectedly but weak enough that a hard yank' triggers it — explicit false-positive engineering. USBKill has a documented vulnerability (Issue #36): 'every device is whitelisted during (re)boot,' so an attacker's device left attached during reboot gets auto-trusted — a trust-on-boot flaw. Lessons for ESP32: (a) tune any physical trigger (button/switch/tether) against accidental activation and require a deliberate gesture (hold-duration, multi-press, or breakaway force), (b) never auto-trust whatever is present at boot — establish the trusted set during a controlled provisioning step, not opportunistically, (c) debounce and require a sustained/confirmed condition before acting.

Sources: https://github.com/hephaest0s/usbkill/issues/36, https://www.buskill.in/, https://hothardware.com/news/buskills-usb-dead-mans-switch-can-make-your-laptop-self-destruct-if-stolen

**[high] Threat model is theft / border-seizure / coercion of a powered-or-locked single device; the protection is data confidentiality, not device denial.**

USBKill was created in response to the Ross Ulbricht arrest (agents grabbed the unlocked laptop before he could lock it) — the canonical 'seized while unlocked' threat. Kali Nuke targets the travel/border-seizure case ('brick laptops before travel... no one is able to restore the data, including ourselves' without the offline key). GrapheneOS duress targets coercion ('hand over your phone and unlock it'). BusKill targets physical snatch-and-grab. Tails targets forensic trace-avoidance and supports emergency shutdown by yanking the USB (RAM wiped on power loss / medium removal). Common thread: the adversary gains PHYSICAL possession; defense is rendering DATA inaccessible. Note: these defend a single owner's own device and data only.

Sources: https://github.com/hephaest0s/usbkill, https://www.kali.org/blog/nuke-kali-linux-luks/, https://tails.net/contribute/design/memory_erasure/, https://en.wikipedia.org/wiki/BusKill

**[high] Hardware (hard) kill switches beat software switches for assurance — sever the circuit, don't politely ask.**

Purism Librem uses DPDT hardware switches (ON-ON, six leads) in the hinge that physically cut power to camera/mic/radios, plus a switch to write-protect BIOS/EC firmware on the Librem 14. Stated principle: 'Unlike a software switch that asks the hardware to turn off politely and hopes it listens, hardware kill switches sever the circuit at the hardware level.' ESP32 implication: where the design relies on a physical arming or trigger, prefer a real hardware switch/jumper (especially for the ARM-the-destructive-tier control and for write-protecting the device's own firmware/config) over a software-only flag that malware or a coercer's forensic tool could flip. Also consider tamper-evidence.

Sources: https://puri.sm/learn/hardware-kill-switches/, https://puri.sm/posts/hard-not-soft-kill-switches/, https://puri.sm/posts/librem-now-most-secure-laptop-under-full-user-with-tamper-evident-features/

**[high] Plausible-deniability (hidden volume) is attractive but fragile and is a documented anti-pattern unless threat-justified.**

VeraCrypt offers hidden volumes/hidden OS that look like random data with no signature, BUT: research ('Defeating Plausible Deniability of VeraCrypt Hidden Operating Systems') shows cross-drive analysis can prove a hidden OS exists and estimate its size from a single image; VeraCrypt's own UI labels a mounted decoy as type 'Outer,' tipping off an aware adversary; and third-party software leaks existence via temp files/thumbnails/cloud sync. Lesson: a 'nothing here' deniability story can be worse than honest crypto-erase, because a sophisticated adversary who detects the decoy may escalate coercion. For most owner-only threat models, prefer simple, honest 'data is gone' crypto-erase over deniability theater unless the user's specific threat model truly warrants it.

Sources: https://veracrypt.io/en/Plausible%20Deniability.html, https://www.researchgate.net/publication/318155607_Defeating_Plausible_Deniability_of_VeraCrypt_Hidden_Operating_Systems, https://github.com/veracrypt/VeraCrypt/issues/1328

**[high] LEGAL/ETHICAL framing is mandatory: owner-only defensive data protection, NOT evasion of lawful process.**

Destruction of evidence (physically destroying, altering, concealing, or falsifying records/electronic data 'pertinent to litigation, investigations, or regulatory matters') is a serious offense; charges can include obstruction of justice, computer crime, and destruction of evidence, with penalties from up to a year for misdemeanors to up to 20 years for federal charges. Many jurisdictions require proof of both destruction AND intent to obstruct. The feature must be documented and gated as protecting the OWNER'S OWN single device against theft/coercion, with an explicit warning that the user must understand evidence-destruction/obstruction laws in their jurisdiction and must not use it to defeat a lawful order or pending proceeding. Recovery/escrow (owner can restore from offline backup) reinforces the 'protection, not spoliation' posture.

Sources: https://oceanliteracydialogues.com/what-is-destruction-of-evidence-legal-definitions-and-consequences/, https://leppardlaw.com/federal/obstruction/electronic-data-manipulation-evolving-challenges-in-digital-evidence-cases/, https://wadelitigation.com/legal-repercussions-of-evidence-destruction-in-civil-litigation/

**[high ⚠HIGH-RISK] Speed and uninterruptibility of the wipe are real requirements, but they directly conflict with safety — resolve by gating, not by slowing the wipe.**

GrapheneOS makes the duress wipe uninterruptible and rebootless on purpose; BusKill prioritized cutting RAM power ASAP over thorough overwrite. The wipe itself should be fast and non-abortable ONCE legitimately triggered (otherwise a coercer interrupts it). Therefore safety MUST live in the layers BEFORE the trigger fires (arming, distinct credential, precedence of legit credential, confirmation only at config-time, debounced/forceful trigger, TEST mode) rather than as an 'are you sure?' at fire-time. Crypto-erase makes fast+permanent feasible because only the key is destroyed.

Sources: https://www.androidauthority.com/grapheneos-duress-pin-3584795/, https://www.buskill.in/luks-self-destruct/, https://www.rebeltech.co.nz/grapheneos-duress-password/

### Recommendations
- REQ-1 (fail-safe default): Default state is DISARMED and least-destructive (lock/shutdown). A device that is freshly flashed, un-provisioned, or in an unknown state must be physically incapable of crypto-erasing real data.
- REQ-2 (crypto-erase architecture): Implement 'wipe' as destruction of the small key/header material the boot-gate holds or wraps — never bulk overwrite. Follow NIST SP 800-88 Rev.2 CE: data must be encrypted with >=128-bit strength for CE to be valid; verify the key is truly gone.
- REQ-3 (explicit two-factor to destroy): Require BOTH explicit ARMING and the trigger condition before any destructive action. Neither arming alone nor a stray trigger alone may destroy data.
- REQ-4 (legit credential precedence): If the destructive trigger is a credential (duress PIN/passphrase), the legitimate credential ALWAYS wins on collision and never wipes; make the duress credential structurally distinct (e.g., different length) to avoid accidental entry.
- REQ-5 (SAFE/TEST mode): Ship a mandatory dry-run mode that exercises detection -> arming-check -> trigger -> erase against a DUMMY key, logging the simulated action, so owners validate wiring/timing/conditions with zero risk. Clearly distinguish SAFE vs ARMED/LIVE in UI and logs.
- REQ-6 (owner-only recovery/escrow): Provisioning must produce an encrypted, OFF-device backup of the key/header (the Kali luksHeaderBackup pattern) that only the owner controls; provide a documented restore path. Never store the recovery secret on the protected device.
- REQ-7 (rate-limit before wipe): Default response to wrong attempts is exponential-backoff lockout (key-brute-force defense), NOT immediate wipe. Any 'N-wrong-attempts -> erase' behavior is a separate, opt-in, ARMED-only mode with a sensibly large N and a monotonic counter that persists across reboots/power-cycles.
- REQ-8 (robust trigger): Debounce and require a sustained/deliberate trigger (hold-duration, multi-press, or tuned breakaway force) to defeat false positives. Make destructive arming a HARDWARE switch/jumper where feasible (Purism hard-switch principle), and write-protect the device's own firmware/config.
- REQ-9 (no trust-on-boot): Establish the trusted device/credential set only during a controlled provisioning step. Never auto-trust whatever is connected/present at boot (USBKill Issue #36 flaw).
- REQ-10 (config-time confirmation, fire-time uninterruptible): Put all 'are you sure?' confirmation, double-entry, and warnings at CONFIGURATION time. Once legitimately triggered, the wipe is fast and non-abortable (coercion resistance). Safety lives before the trigger, not at fire-time.
- REQ-11 (observability): Surface arming state and SAFE/LIVE mode via an indicator (LED) and persistent log so the owner always knows whether the device is hot.
- REQ-12 (legal/ethical gating + docs): Frame and document strictly as owner-only, single-device, defensive data-confidentiality protection against theft/border-seizure/coercion. Include an explicit notice that the user must understand evidence-destruction and obstruction-of-justice laws in their jurisdiction and must not use it to defeat lawful process; reinforce the protective (recoverable, owner-escrowed) posture over a spoliation posture.

### Open questions
- ANTI-REQ candidates to confirm with the user: (a) NO bulk-overwrite/zeroization of full storage (slow, defeats the crypto-erase model); (b) NO plausible-deniability/hidden-volume feature unless the user's threat model specifically demands it (fragile, detectable, can escalate coercion); (c) NO destructive-by-default shipping config; (d) NO on-device storage of the recovery/escrow secret; (e) NO small/low default for any N-strikes wipe counter; (f) NO trust-on-boot of present devices.
- What exactly does the ESP32 boot-gate control — does it physically hold/release a disk-unlock key (HSM-style key custody), gate power/boot to a host, or store the wrapped LUKS/volume key? This determines whether crypto-erase = wiping ESP32-resident key material vs sending a luksErase-style command to a host.
- Is the trigger credential-based (duress PIN at a prompt), physical (dead-man tether/button), absence-based (dead-man timeout), or attempt-counter-based? Each has different fail-safe defaults and false-positive profiles.
- Will the device ever be in scope of a lawful preservation order or be a shared/multi-user device? If so, the legal framing and possibly the feature availability need revisiting.
- What is the desired wipe latency and power model (does the ESP32 retain power to complete the key destruction, or must it survive a yanked-power/snatch scenario like BusKill/Tails)? This affects whether key material must be in fast-erasable storage.
- Should the design include tamper-evidence / firmware write-protection (Purism-style) so a forensic actor cannot reflash the gate to bypass it before the owner notices?

### Adversarial verifications
- **[CONFIRMED]** Crypto-erase (destroy the key/header, not the data) is the universal mechanism and the formal modern secure-erase standard.
  - Every load-bearing technical sub-claim checks out against primary sources; the only inaccuracy is rhetorical overreach in the word "universal."

NIST SP 800-88 (verified against the actual NIST PDF text I extracted from the r2 Initial Public Draft, plus reliable secondary reporting of the FINAL r2 published 2025-09-26):
- Section 3.2 ("Use of Cryptography and Cryptographic Erase") states verbatim: "Cryptographic erase (CE) leverages the encryption of target data by enabling sanitization of the target data's encryption key. This leaves only the ciphertext remaining on the ISM, effectively sanitizing the data by preventing read-access. Without the encryption key, the target data is unrecoverable." => CONFIRMS "destroy the key, not the data."
- "with CE, sanitization may be performed with high assurance much faster than with other sanitization techniques" and "Typically, CE can be executed in a fraction of a second." => CONFIRMS "high assurance far faster than overwrite."

## Host-side ESP32 flasher: multi-image Guardian+Marauder flash + per-device config provisioning (esptool/esp-idf), integrated into headless-marauder-gui

**Summary.** A host-side Python flasher wrapping esptool can do everything the design needs in ONE provisioning pass: esptool write_flash accepts an arbitrary list of `<offset> <file>` pairs in a single invocation, so the existing flasher.flash() already has the exact shape needed — it just needs a longer files[] list that adds the Guardian (factory) app, a custom partition table, an NVS config blob, and a fresh otadata image. Per-device config is baked by generating an NVS partition binary from a CSV with esp-idf's nvs_partition_gen.py (arm GPIO pin, armed level, max attempts, wipe flags as u8/u32 entries; password stored ONLY as a salted KDF hash + salt as hex2bin/binary blobs). The password hash MUST be computed on the host at flash time (Argon2id/PBKDF2 over the user-typed password + a fresh random salt) and only the {salt, hash, KDF params} written into the CSV — never plaintext, never committed. First boot is forced into Guardian by writing an erased (0xFF) otadata image; the stock Marauder partition layout has NO factory partition, so this requires a CUSTOM partition table (factory=Guardian + ota_0=Marauder). At-rest protection uses ESP32 flash encryption (host burns/option-generates the key; device encrypts in place on first boot) — but note flash encryption does NOT cover the NVS partition, which needs separate NVS encryption. esptool auto-verifies every written file (post-write read-back + MD5, reflash on mismatch); image_info/verify_flash/read_flash give explicit verification. The minimal UX is a single "Suicide" checkbox in the existing per-hardware flow that swaps the plain Marauder app for the Guardian+Marauder bundle (custom partition table + factory + ota_0 + NVS + otadata) and prompts only for password and arm pin/level.

### Findings

**[high] esptool write_flash already accepts an arbitrary number of <offset> <file> pairs in one call — the existing flasher.flash() is the correct shape; it just needs more pairs appended.**

Docs: 'The next arguments to write-flash are one or more pairs of offset (address) and file name' and 'Multiple flash addresses and file names can be given on the same command line.' The existing <HOME>/projects/headless-marauder-gui/marauder_core/flasher.py builds exactly this: it constructs files[] by iterating support.items() (offset->path) then appends ['0x10000', app_path] and calls esptool_argv('--chip',chip,'--port',port,'--baud',...,'write_flash','-z','--flash_size','detect',*files). A Guardian+Marauder pass just extends that list, e.g.: 0x1000 bootloader.bin, 0x8000 custom_partitions.bin, 0xe000 boot_app0.bin, 0x10000 guardian_factory.bin, <ota_0 off> marauder.bin, <nvs off> nvs_config.bin, <otadata off> otadata_erased.bin — all in one write_flash. The '-z' (compress, default on; '-u/--no-compress' disables) and '--flash_size detect' flags are already used and should be kept.

Sources: https://docs.espressif.com/projects/esptool/en/latest/esp32/esptool/basic-commands.html, <HOME>/projects/headless-marauder-gui/marauder_core/flasher.py

**[high ⚠HIGH-RISK] The stock ESP32 Marauder partition table has NO factory partition (it is an OTA-style layout), so a dual Guardian+Marauder build REQUIRES a custom partition table that adds a factory slot. This is the load-bearing structural change.**

Marauder ships its own esp32_marauder.ino.partitions.bin (flasher.py fetches it from FlashFiles/<dir>/ at 0x8000). Typical Arduino/Marauder big-app layouts (and esp-idf default_8MB) use nvs+otadata+app0(ota_0)+app1(ota_1)+spiffs with NO factory partition. To boot Guardian first and keep Marauder as the swappable payload, define a CUSTOM table: nvs (data/nvs), otadata (data/ota, 0xe000 size 0x2000), factory (app/factory at 0x10000) = Guardian, ota_0 (app/ota_0) = Marauder, optional spiffs. Docs: 'The factory partition subtype is the default application. The bootloader will set this as the default if no OTA partition is found, or the OTA partitions are empty.' The flasher must write this custom partitions.bin at 0x8000 INSTEAD of Marauder's stock one when Suicide mode is on. Sizing: Marauder app is large (~3MB+), Guardian is tiny, so a 16MB module is comfortable; on 4MB you must shrink/verify slot sizes.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-guides/partition-tables.html, https://docs.espressif.com/projects/arduino-esp32/en/latest/tutorials/partition_table.html, <HOME>/projects/headless-marauder-gui/marauder_core/flasher.py

**[high ⚠HIGH-RISK] Forcing first boot into Guardian (factory) is done by flashing an ERASED (all 0xFF) otadata image; the 2nd-stage bootloader boots factory when otadata is blank.**

Docs: 'When the otadata partition contains no data (all bytes erased to 0xFF), the ESP-IDF second stage bootloader boots the factory app if it is present... If no factory app is included, the first available OTA slot (usually ota_0) is booted.' Two host options: (a) include an offset/file pair that writes a 0x2000 file of 0xFF at the otadata offset (0xe000 in default layouts) inside the single write_flash; or (b) after flashing, run esp-idf's otatool.py (e.g. `otatool.py -p PORT erase_otadata`). Option (a) keeps it in the one-shot esptool call and needs no esp-idf install — generate the 0xFF blob on the host (size = otadata partition size). This guarantees Guardian/factory runs on first power-up regardless of any stale OTA state, which is exactly the boot-gate requirement.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/system/ota.html, https://github.com/espressif/esp-idf/blob/master/components/app_update/otatool.py

**[high] Per-device config (arm GPIO pin, armed level, max attempts, wipe options) is baked by generating an NVS partition binary from a 4-column CSV with esp-idf's nvs_partition_gen.py, then flashing it at the nvs offset.**

CSV header is exactly `key,type,encoding,value`. First row must be a namespace row: `guardian,namespace,,`. Then config rows, e.g. `arm_gpio,data,u8,12` / `armed_level,data,u8,1` / `max_attempts,data,u8,3` / `wipe_flags,data,u32,7`. Supported data encodings: u8,i8,u16,i16,u32,i32,u64,i64,string,hex2bin,base64,binary (string is NUL-terminated, binary is not). Generate: `python nvs_partition_gen.py generate config.csv nvs_config.bin 0x3000` (size multiple of 4096, min 0x3000) — partition size must match the nvs partition size in the custom table. Then add `<nvs_offset> nvs_config.bin` to the write_flash pair list. The Guardian firmware reads these keys from the 'guardian' namespace at boot.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/storage/nvs_partition_gen.html

**[high ⚠HIGH-RISK] The password is stored ONLY as a salted KDF hash + salt (+ KDF params) in NVS — never plaintext, never in the repo. The hashing happens on the HOST at flash time, in the flasher, immediately before generating the NVS CSV.**

Flow: user types password in the UI -> host generates a fresh random salt (e.g. 16 bytes from os.urandom) -> host computes hash = Argon2id(password, salt, m/t/p params) (or PBKDF2-HMAC-SHA256 with high iteration count if you must avoid native deps on the cyberdeck) -> write to CSV: `pw_salt,data,hex2bin,<salt_hex>`, `pw_hash,data,hex2bin,<hash_hex>`, plus params like `pw_kdf,data,string,argon2id` / `pw_m,data,u32,19456` / `pw_t,data,u8,2` / `pw_p,data,u8,1`. The plaintext password is held only in a local variable and zeroized/discarded; it is never written to disk, never logged (note flasher.py streams the esptool argv to the console — the NVS bin path is fine but DO NOT echo the password or pass it as a CLI arg). Guardian re-derives the hash from the entered passphrase + stored salt/params and compares to pw_hash to unlock. OWASP minimum for Argon2id: m=19456 (19MB), t=2, p=1; for a 240MHz ESP32 with limited RAM you will likely tune Argon2 memory DOWN to fit (~tens of KB) and accept the device-side verify cost — confirm the on-device KDF can allocate the chosen m. PBKDF2 is the safe low-memory fallback for the constrained device.

Sources: https://guptadeepak.com/research/password-hashing-guide-2026/, https://en.wikipedia.org/wiki/Argon2, https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/storage/nvs_partition_gen.html

**[high ⚠HIGH-RISK] At-rest protection uses ESP32 flash encryption: the host enables it / optionally pre-burns the AES key, and the device encrypts flash in place on first boot. Use Release mode for real protection.**

Two host approaches: (1) Default/easiest — flash plaintext images, and on first boot the 2nd-stage bootloader generates a random AES-256 key into the flash_encryption eFuse (read/write protected, not software-accessible) and encrypts partitions in place (can take ~1 min). (2) Host-controlled key — `idf.py secure-generate-flash-encryption-key key.bin` then `idf.py --port PORT efuse-burn-key flash_encryption key.bin` (burnable once, BEFORE first encrypted boot), enabling host pre-encryption via `idf.py secure-encrypt-flash-data --keyfile key.bin --address 0x10000 --output ct.bin pt.bin` flashed with esptool. FLASH_CRYPT_CNT eFuse tracks state (odd bit-count = encrypted). Development mode allows re-flashing plaintext (up to 3 toggles) for testing; Release mode write-protects FLASH_CRYPT_CNT and sets DISABLE_DL_DECRYPT/CACHE — choose Release for shipped anti-forensic devices. WARNING: flash encryption is a ONE-WAY eFuse burn — bricks-on-mistake risk and no plaintext re-flash in Release; gate it behind an explicit, separate confirmation, NOT the same Suicide checkbox.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/security/flash-encryption.html

**[high ⚠HIGH-RISK] Flash encryption does NOT encrypt the NVS data partition — the config + password hash blob needs SEPARATE NVS encryption, or you accept that the salted hash (not plaintext) is the only thing exposed.**

Docs: 'The nvs partition cannot be encrypted, because the NVS library is not directly compatible with flash encryption.' Flash encryption auto-covers bootloader, partition table, otadata, the NVS-KEY partition (metadata), and all app partitions — but the NVS data partition itself is left plaintext. To encrypt the config blob, use NVS encryption: `python nvs_partition_gen.py encrypt config.csv nvs_config.bin 0x3000 --keygen --keyfile nvs_keys.bin` (XTS-AES), add an `nvs_keys` partition (subtype nvs_keys, flagged encrypted so flash encryption protects the NVS key), and flash both. Mitigating factor for the threat model: because the password is stored as a salted KDF hash (not plaintext), even an unencrypted NVS dump does not reveal the passphrase; NVS encryption mainly hides the arm-pin/level/wipe config and raises the bar for tamper. Decide based on threat model whether NVS encryption is worth the extra key partition + complexity.

Sources: https://docs.espressif.com/projects/esp-idf/en/latest/esp32/security/flash-encryption.html, https://docs.espressif.com/projects/esp-idf/en/latest/esp32/api-reference/storage/nvs_partition_gen.html

**[high] Verification of flashed images is partly automatic and partly explicit. esptool auto-verifies every written file; explicit verification uses image_info, verify_flash, and read_flash.**

Auto: 'esptool always verifies the data after write and automatically reflashes the whole file if verification fails' (post-flash MD5 / read-back of changed sectors). App images carry a SHA256 digest in the extended header (hash_appended=0x01) that the bootloader checks at boot. Explicit host-side checks for the flasher to surface: `esptool image_info <app.bin>` (load addresses, segment sizes, flash size/freq/mode, SHA256) to sanity-check each bundle file BEFORE flashing; `esptool verify_flash <offset> <file>` to re-verify after; `esptool read_flash <addr> <size> out.bin` (size accepts k/M/ALL) to dump for audit. Practically, rely on esptool's built-in post-write verify for the flash itself and add an image_info pre-check so a wrong-chip/corrupt bundle file is caught before the eFuse-burning encryption step.

Sources: https://docs.espressif.com/projects/esptool/en/latest/esp32/esptool/basic-commands.html, https://docs.espressif.com/projects/esptool/en/latest/esp32/advanced-topics/firmware-image-format.html

**[high] Minimal flasher UX: a single 'Suicide' checkbox in the existing per-hardware flow swaps the plain Marauder app for the Guardian+Marauder bundle and prompts only for password + arm pin/level.**

Concrete integration into <HOME>/projects/headless-marauder-gui/gui/flasher_window.py: add one ttk.Checkbutton 'Suicide (Guardian boot-gate)' next to the existing Mode radios. When ticked: (1) reveal a tiny sub-form with exactly two prompts — Password (masked Entry) and Arm pin + Armed level (a GPIO spinbox + High/Low toggle); everything else (max_attempts, wipe_flags) defaults sensibly and stays hidden. (2) The chip detection, port, baud, and 'Load release list' flow are UNCHANGED — the selected Marauder release .bin becomes ota_0 instead of the 0x10000 app. (3) On FLASH: hash the password on the host (see hashing finding), run nvs_partition_gen generate to make nvs_config.bin, generate the 0xFF otadata blob, fetch/include the custom partition table + Guardian factory bin (ship these in the repo or a release, like the existing FlashFiles fetch), then call a new flasher.flash_suicide() that appends all pairs to ONE write_flash. Reuse the existing _worker/_run_stream/_free_port plumbing verbatim. ## ESP32 Marauder boot-password + arming-switch + secure-wipe UX feasibility, per hardware input class

**Summary.** Marauder's firmware already ships three on-device text/selection primitives that decide everything: HAS_MINI_KB (an on-screen grid keyboard driven by buttons/joystick), the touch keyboard (HAS_TOUCH/HAS_CYD_TOUCH), and the Cardputer physical QWERTY. Any board with a screen plus ANY input (joystick, 2 buttons, touch, or keyboard) can therefore host a fully self-contained on-device password prompt and wipe-confirm UI; truly headless boards (Flipper Dev Board Pro, Lonely Binary Gold, bare dev boards) have no screen and must gate over the USB-serial CLI from the host (headless-marauder-gui). The hard constraint is GPIO, not UX: on the all-in-one boards (CYD, Mini, M5StickC, Cardputer) almost every pin is consumed by display+touch+SD+keyboard-matrix, and the few free pins are frequently input-only (GPIO34-39), which is actually fine for a read-only arming switch but leaves little for anything else. Recommended split: full on-device gate for touchscreen (b), joystick Mini (c), and Cardputer (d); on-device gate with a compromised/short PIN for M5StickC (e, only 2-3 buttons); serial/host-assisted gate only for headless (a). Self-destruct (NVS erase + SPIFFS/SD overwrite + esp_partition wipe) is standalone-capable on every class because it is pure firmware acting on internal flash; the SD card is the only externally-removable evidence and can only be overwritten while inserted.

### Findings

**[high ⚠HIGH-RISK] Marauder firmware already contains on-device text-entry primitives for every input type, so a self-hosted password prompt is feasible on any board that has a screen plus any input.**

configs.h / menu system define three input keyboards: HAS_MINI_KB = an on-screen grid keyboard (alphanumeric + Backspace/Space/Done) navigated by buttons/joystick for button-only devices; a full Touch Keyboard for HAS_TOUCH/HAS_CYD_TOUCH devices; and the Cardputer physical QWERTY where ';'=Up '.'=Down '('=Select. HAS_MINI_KB is enabled for MARAUDER_MINI, V6, V7, Sunton35 and CYD-class builds. The v7 also exposes a user-facing on-screen keyboard for Ducky script editing, proving free-text entry already works on-device. A PIN/passphrase prompt is just a reuse of these existing widgets.

Sources: https://deepwiki.com/justcallmekoko/ESP32Marauder/2.4-menu-system, https://github.com/justcallmekoko/ESP32Marauder/wiki/run-ducky-script, https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h

**[high ⚠HIGH-RISK] (a) Headless / serial-only boards have no screen and no usable nav input, so the password gate MUST be host/serial-assisted.**

Supported headless targets: Flipper Zero WiFi Dev Board (ESP32-S2), Flipper MultiBoard/Dev Board Pro S3 (ESP32-S3), Lonely Binary Gold / bare ESP32 dev boards, ESP32-C5 DevKit. In configs.h these define neither HAS_SCREEN nor HAS_BUTTONS. Interaction is exclusively the UART/USB CLI: getSerialInput() -> Serial.readStringUntil('\n') -> runCommand() dispatch (commands like scanap, attack, reboot, settings, save/load). A boot password therefore can only be a serial 'unlock <pass>' command typed from the host (headless-marauder-gui / web-serial), OR a crude physical button-sequence on the single BOOT/EN button if present. No standalone on-device prompt is possible.

Sources: https://deepwiki.com/justcallmekoko/ESP32Marauder/5.1-supported-boards-and-variants, https://github.com/justcallmekoko/ESP32Marauder/wiki/cli, https://deepwiki.com/justcallmekoko/ESP32Marauder/2.3-command-line-interface, https://github.com/justcallmekoko/ESP32Marauder/wiki/Headless-Mode

**[high] (b) Touchscreen boards (CYD 2.4-3.5", Marauder v6/v7/v8) can run a full standalone on-device PIN/keypad gate via the existing touch keyboard.**

CYD builds set HAS_TOUCH + HAS_CYD_TOUCH; v6/v7 set HAS_TOUCH. The touch keyboard widget already exists, so an on-screen PIN pad or alphanumeric passphrase entry is a direct reuse. No host needed. Touch and SD share the VSPI bus with separate CS lines, so a wipe routine can drive the screen and erase the SD sequentially.

Sources: https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h, https://randomnerdtutorials.com/esp32-cyd-display-touchscreen-microsd-card/, https://github.com/justcallmekoko/ESP32Marauder/wiki/run-ducky-script

**[high] (c) Marauder Mini (5-way joystick) can run a full standalone on-device gate via HAS_MINI_KB directional code entry.**

configs.h MARAUDER_MINI block: L_BTN=13, C_BTN=34, U_BTN=36, R_BTN=39, D_BTN=35, with HAS_L/R/U/D/C all enabled and HAS_MINI_KB defined. User scrolls the on-screen grid keyboard with the joystick and selects with center -> full passphrase, or a fast directional combo (e.g. U-U-D-L-R-C) as a PIN. Fully self-hosted, no serial needed.

Sources: https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h, https://github.com/justcallmekoko/ESP32Marauder/wiki/marauder-versions

**[high] (d) M5Cardputer can run the strongest standalone gate: real QWERTY passphrase entry.**

MARAUDER_CARDPUTER (ESP32-S3) keyboard maps ';'=Up '.'=Down '('=Select for menu nav, but the firmware reads the full key matrix, so an arbitrary-length alphanumeric password is entered natively. Original Cardputer scans a 74HC138-demux matrix on G3/4/5/6/7/8/9/11/13/15; Cardputer ADV offloads to a TCA8418 on I2C (GPIO8/9 + INT GPIO11). C_BTN=0 (BOOT) is the only discrete 'button'. Best UX of all classes for a strong passphrase.

Sources: https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h, https://docs.m5stack.com/en/core/Cardputer, https://github.com/RetroBreeze/cardputer-keyboard-reference, https://deepwiki.com/justcallmekoko/ESP32Marauder/2.4-menu-system

**[high ⚠HIGH-RISK] (e) M5StickC Plus can self-host a gate but only as a weak button-combo code; the original Plus exposes only 2 usable buttons.**

configs.h: M5STICKC C_BTN=37, D_BTN=39 (U_BTN=-1) -> 2 buttons; M5STICKCP2 adds U_BTN=35 -> 3 buttons. HAS_MINI_KB is NOT defined for the StickC builds, so there is no on-screen grid keyboard; a passphrase is impractical. Realistic gate = a Morse/combo sequence on the 2-3 buttons (e.g. short-short-long-A-B). Standalone is possible but low entropy; consider a longer sequence or a host-serial fallback for a real password. Note GPIO37/39 are input-only, which is fine for buttons.

Sources: https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h, https://docs.m5stack.com/en/core/m5stickc_plus, https://community.m5stack.com/topic/1283/m5stick-c-button-pin-mapping

**[high ⚠HIGH-RISK] Arming-switch GPIO placement is the real bottleneck; on the all-in-one boards almost every pin is consumed and the survivors are often input-only (GPIO34-39) — acceptable for a read-only switch but nothing else.**

CYD (ESP32-2432S028R): display=GPIO2/12/13/14/15+BL21, touch=GPIO25/32/33/36/39, SD=GPIO5/18/19/23, RGB LED=GPIO4/16/17, LDR=GPIO34, audio=GPIO26. Free on the P3/CN1 headers: GPIO22 and GPIO27 (full I/O; GPIO27 has pull-up R18) and input-only GPIO35. An arming switch goes best on GPIO22 or GPIO27, or read-only on GPIO35. Mini/v7 joystick already eats 13/34/35/36/39, leaving little. M5StickC: Grove HY2.0 exposes G32/G33 (best for arming switch); G0 is mic-CLK, G26/G36 share a port (set the unused one floating). Cardputer: Grove HY2.0 exposes G1/G2 — use one for arming. Headless dev boards have many free pins, so an arming GPIO is trivial there (but the gate is still serial).

Sources: https://randomnerdtutorials.com/esp32-cheap-yellow-display-cyd-pinout-esp32-2432s028r/, https://www.esp32s.com/blog/the-complete-esp32-cheap-yellow-display-cyd-pinout-and-gpio-guide/, https://docs.m5stack.com/en/core/m5stickc_plus, https://docs.m5stack.com/en/core/Cardputer, https://raw.githubusercontent.com/justcallmekoko/ESP32Marauder/master/esp32_marauder/configs.h

**[medium ⚠HIGH-RISK] Standalone self-destruct is feasible on EVERY class because the wipe acts on internal flash via firmware; only the removable SD card is an evidence gap.**

A wipe routine = nvs_flash_erase() (clears saved configs/creds), SPIFFS/LittleFS format, esp_partition_erase_range over data partitions, and optional overwrite-then-erase passes; none of this needs a screen or host. The Marauder CLI already has reboot/save/load, so adding an internal 'wipe' command/menu action is straightforward. Caveat: ESP32 NOR flash has no guaranteed secure-erase/wear-leveling transparency, so 'overwrite then erase' is best-effort. The microSD shares VSPI with touch and can be overwritten only while inserted; a pulled card cannot be wiped. Recommend encrypting SD contents at rest so a wipe of the on-flash key is sufficient. Enabling ESP32 flash encryption + secure boot makes a power-off state itself resistant to readout.

Sources: https://randomnerdtutorials.com/esp32-cyd-display-touchscreen-microsd-card/, https://github.com/justcallmekoko/ESP32Marauder/wiki/cli, https://deepwiki.com/justcallmekoko/ESP32Marauder/2.3-command-line-interface

**[high] Headless boards can still get a hardware arming switch and standalone wipe-on-trigger even without a password UI.**

Bare ESP32/S2/S3 dev boards and the Flipper Dev Board Pro expose many free GPIOs, so a physical arming toggle + a 'panic' momentary button can be wired directly and the firmware can wipe on GPIO trigger with no host attached. What they cannot do standalone is PROMPT for and verify a typed password — that gate must come over serial. So: arming-switch + wipe = yes standalone; password ENTRY = serial/host only.

Sources: https://deepwiki.com/justcallmekoko/ESP32Marauder/5.1-supported-boards-and-variants, https://github.com/justcallmekoko/ESP32Marauder/wiki/Headless-Mode

### Recommendations
- TIERED FEATURE MATRIX (class -> password input | arming-switch feasible? | standalone self-destruct? | notes):
- (a) Headless/serial-only (Lonely Binary Gold, bare ESP32/S2/S3 dev boards, Flipper Dev Board Pro): password = HOST/SERIAL only ('unlock <pass>' via headless-marauder-gui) or crude BOOT-button sequence | arming switch = YES, trivial (many free GPIOs) | standalone self-destruct = YES (flash wipe needs no host; can trigger on GPIO) | note: cannot prompt/verify a typed password without the host -> SERIAL-GATE / HOST-ASSISTED.
- (b) Touchscreen (CYD 2.4-3.5", Marauder v6/v7/v8): password = on-screen touch PIN/keypad or passphrase (reuse existing touch keyboard) | arming switch = YES on CYD free GPIO22/27 (or read-only GPIO35) | standalone self-destruct = YES | note: best balance -> FULL ON-DEVICE GATE.
- (c) Joystick (Marauder Mini): password = directional combo or HAS_MINI_KB grid passphrase (joystick L13/C34/U36/R39/D35) | arming switch = HARD (joystick eats 13/34/35/36/39; use Grove/free pin or repurpose, mostly input-only survivors) | standalone self-destruct = YES | note: FULL ON-DEVICE GATE, but arming-switch GPIO is tight.
- (d) M5Cardputer: password = native QWERTY passphrase (strongest entropy of all classes) | arming switch = YES via Grove HY2.0 G1/G2 | standalone self-destruct = YES (ESP32-S3) | note: FULL ON-DEVICE GATE, preferred for strong passphrases.
- (e) M5StickC Plus: password = button-combo / Morse code only (2 buttons on Plus: G37/G39; 3 on Plus2 +G35; no HAS_MINI_KB) | arming switch = YES via Grove G32/G33 | standalone self-destruct = YES | note: low-entropy gate -> ON-DEVICE GATE (WEAK) or pair with serial fallback for real passwords.
- Verdict: FULL on-device gate = touchscreen (b), Cardputer (d), Mini (c). On-device gate but WEAK (button-combo) = M5StickC (e). SERIAL-GATE / host-assisted only = headless (a).
- Implement the password prompt as a reuse of the existing keyboard widgets (touch keyboard / HAS_MINI_KB / Cardputer matrix) gated by a compile-time INPUT_CLASS flag mirroring HAS_TOUCH / HAS_MINI_KB / MARAUDER_CARDPUTER / HAS_BUTTONS / headless, so one codebase covers all five classes.
- For the arming GPIO, add a per-board #define ARMING_PIN with safe defaults: CYD=GPIO22 or 27, M5StickC=G32, Cardputer=G1, headless=any free pin; for Mini fall back to a Grove/exposed pin since the joystick consumes the input-only block. Treat input-only pins (34-39) as legitimate for a read-only switch.
- Make self-destruct act on internal flash (nvs_flash_erase + partition erase/overwrite) so it is standalone on every class; encrypt SD contents at rest and enable ESP32 flash-encryption + secure-boot so wiping the on-flash key is a sufficient and fast panic action, and a pulled/powered-off device is not trivially readable.
- On headless boards, expose BOTH a serial 'unlock'/'wipe' command AND an optional hardware arming-toggle + panic button so the device can self-destruct in the field with no laptop attached, while the password verification remains a host-side serial step.
